# 引き継ぎ: Stage 3c の厳密Nashテーブル作成とweb配信

## ゴール

Stage 3b と**同じ手法**(幾何打ち切りゼロサム確率ゲームの厳密な定常Nashを、キャッシュ版
backward VI で解く)で 3c のポリシー/勝率テーブルを作り、web で対戦できるようにする。

## 前提: 3b で確立済みの手法(そのまま流用可)

3b は完了済み(コミット `a80f0ed`〜`4602e88`、いずれも**未push**)。要点:

- **ゲームモデル**: 毎ターン確率 `1-γ=0.01` で対戦終了(メモリレスな幾何打ち切り)、終了時は
  残HPタイブレーク(生存数→残HP割合)で決着。これでShapleyの定常均衡が存在し、`turn` を状態に
  持たない定常テーブルが厳密Nashになる(100手引き分けだとstall縮退で解けない、が結論)。
- **ソルバ**: `poke-ai3-python/src/nash_vi_cache.rs` の `solve_nash_cached`(pyo3)。遷移分布を
  CSR平坦配列に前計算して各反復で使い回すキャッシュ版。H=26で約4分・0.6GB。
- **手法解説**: `web/method.html`(日英)。exploit分析: `experiments/poke-ai3/20260715_1142_*.md`。

## 重要: 3c はソルバ無改造で通る(検証済み)

3c は 3b と**構造が完全に同一**:
- 両側 {Cloyster, 通常Goodra} の 2 体パーティ・各 1 技・クロスチーム(Team1 vs Team2)。
- 違いは技(FightSpe60/FairyPhy60 ↔ ShockWave/Bulldoze)と**対称性**(3c は対称、3b は非対称)。
- dense index の radix も同一(`total=3,655,808`, H=26, cross_team_only, member0=Cloyster/
  member1=Goodra)。既存 `web/policy_3c.meta.json` の `species_order=['Cloyster','Goodra']`。

H=6 で `solve_nash_cached('3c', ...)` を実行して確認済み:
```
3c H=6: exploit mean=0.00022 max=0.00366   ← ≈0(厳密Nash)
start t0 V=0.505 / t1 V=0.495              ← ほぼ0.5/0.5(対称=期待通り。3bは0.472/0.528)
mixed(.02-.98)=1198
```
→ `nash_vi_*.rs` / `solve_nash_cached` は**変更不要**。

## 作業手順

### 1. 本計算(H=26)

`scripts/run_nash_geo_h26.py` を 3c 用に流用する(`stage`・出力パスを '3c' に。または引数化)。
```bash
cd poke-ai3-python
# run_nash_geo_h26.py の solve_nash_cached('3b',...) を '3c' に変え、npz 出力を nash_geo_h26_3c.npz へ
setsid nohup uv run python scripts/run_nash_geo_h26.py > /tmp/psro/nash_geo_h26_3c.log 2>&1 &
# 約4分・0.6GBメモリ。exploit mean~1e-5, max<0.01 を確認。
```
出力(npz)は `data/poke-ai3/nash_geo/`(gitignore)へ。

### 2. 検証

- exploitability: mean≈0、max<0.01。
- 鏡像対称: 各 valid で `V(s)+V(swap(s))≈1`(3c は対称なので開始 V≈0.5)。
- mixed 状態が多数(3bは110万/338万)。縮退(全pure/全0.5)していないこと。

### 3. web形式へ書き出し(★ここだけ実装が要る)

`scripts/export_nash_geo_web.py` は現状**3bハードコード**(`stage='3b'`, `species_order`
`['Cloyster','Goodra-Hisui']`, 出力 `policy_3b.bin`)。3c 用に一般化する:
- 入力 npz を 3c のものに、`stage='3c'`、`species_order=['Cloyster','Goodra']`、出力を
  `web/policy_3c.bin` / `web/value_3c.bin` / `web/policy_3c.meta.json` に。
- radix・sentinel・scale は 3b と同一なので配列はそのまま `tofile`。
- 推奨: export_nash_geo_web.py を stage 引数化して 3b/3c 両対応にする。

現行 `web/policy_3c.bin`(旧手法・単一ネット由来)を上書きする前に scratchpad へバックアップ。

### 4. 整合検証

## deploy 方針(3bと共通)

- web は**ターン上限なし(引き分け打ち切りなし)**でデプロイ(`poke-wasm` で `with_max_turns(0)`)。
  「1%突然終了」も「100手引き分け」も人間に不自然、という判断。
- そのため**stall狙いexploit**(HP劣勢を無限相互交代ループに逃がす)は 3c にも同様に存在する。
  これは幾何打ち切りを外した代償で、意図的に受容済み(`experiments/.../20260715_1142_*.md` 参照)。
  3c でも exploit 度合いを測るなら `best_response_vs_table('3c', 26, policy.tolist(), ...)` が使える。
