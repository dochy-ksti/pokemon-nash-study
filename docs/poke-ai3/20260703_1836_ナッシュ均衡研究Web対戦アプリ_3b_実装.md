# 「ポケモンにおけるナッシュ均衡の研究」Web対戦アプリ (stage 3b) 実装ハンドオフ

実装日: 2026-07-03 (JST)

## 概要

学習済みポリシー AI を GPU 不要の静的Webサイト(GitHub Pages 想定)で遊べるようにした。
全状態のポリシーを事前に GPU 計算してテーブル化し、ブラウザは推論せずテーブル参照だけで
賢い AI と対戦する。バトル進行(乱数・急所ありのダメージ計算/ターン解決)は `poke-sho-rust`
を WebAssembly 化してブラウザ内で厳密に回す。まず stage 3b(ヒスイヌメルゴン vs パルシェン)。

設計合意の詳細は grill-me セッション参照。要点:
- デプロイ = 乱数あり・急所あり・命中100%。HP 4% 離散化(26バケット)。
- **クロスチーム限定**(学習分布と一致)。学習側 [game_task.rs:91](../../poke-ai3/src/game_task.rs#L91) は
  3b で必ず Team1 vs Team2 のみ生成し、同チーム対戦(Team1 vs Team1 等)は学習していない。
  よって Web も相手AIを人間の反対チームに固定し、同チームOOD局面を排除。`opp_team = 1 - ai_team` は
  一意なのでテーブル次元から削除(状態数 8·H^4 = 3,655,808、7.3MB)。
- 埋め込みは素のポリシー。個体 = `data/poke-ai3/tournament/K1b5_ep280.pt`(3b 個体レート最高 +10.8)。
- テーブル = 密 mixed-radix 配列、値は P(交代) を u16(0..1000, 0.1%精度)。1状態1値。

## 成果物

### A. WASM sim crate `poke-wasm/`(新規、ワークスペース member)
- 依存 `poke-sho-rust` + `rand_chacha`(rand/getrandom を回避)+ `wasm-bindgen` + `serde-wasm-bindgen`。
- [lib.rs](../../poke-wasm/src/lib.rs): `Battle`(new / setHp / snapshot / legal / damageRange / step)。
  `step` は乱数・急所ありで `apply_turn`、瀕死強制交代は控え一意で自動解決。
- [view.rs](../../poke-wasm/src/view.rs): JS へ渡す表示ビュー(種族名・実数値・タイプ・技・HP)。
- [rng.rs](../../poke-wasm/src/rng.rs): `BattleChaCha` 相当の最小 RNG(seed は JS 注入)。
- ビルド: `wasm-pack build poke-wasm --target web --out-dir ../web/pkg`(生成物 84KB)。

### B. テーブル前計算(poke-ai3-python)
- PyO3 列挙器 [policy_table.rs](../../poke-ai3-python/src/policy_table.rs):
  `enumerate_policy_batch(stage, hp_buckets, start, count)` が正準列挙順(密index)で有効状態の
  観測を作りエンコード dict + dense index を返す。既存 `encode_observation_states` と同居。
  `poke_ai3.__init__` で re-export。
- ドライバ [export_policy_table.py](../../poke-ai3-python/scripts/export_policy_table.py):
  `Agent(infer_graph=False)` で checkpoint をロード → バッチ推論 → P(交代)=policy[:,4:].sum() を
  u16 で密配列へ → `web/policy_3b.bin` + `web/policy_3b.meta.json`。`make export-policy-table ARGS=...`。
- 生成コマンド(クロスチーム限定、7.3MB、valid 3,380,000 / 3,655,808):
  ```
  cd poke-ai3-python
  make export-policy-table ARGS="--checkpoint ../data/poke-ai3/tournament/K1b5_ep280.pt \
    --stage 3b --hp-buckets 26 --out-dir ../web"
  ```

### C/D. 静的Webアプリ `web/`
- [index.html](../../web/index.html) / [style.css](../../web/style.css) / [app.js](../../web/app.js)。
  CSP/外部依存なしの自己完結。ランディング(チーム・先発選択)→ 同時手番の対戦画面。
- 相手のポリシー・自分のAI推奨手を%併記(個別トグル)、両方向の推定ダメージ min–max%、
  日英切替(種族名・技名・UI)。AI は同視点でテーブルを引き `Math.random` でサンプル。
- テーブル index は JS の `denseIndex` で mixed-radix 計算(meta.json の radix と一致)。

## 検証結果

- `cargo build -p poke-wasm --target wasm32-unknown-unknown` OK / `cargo test --workspace` 全パス
  (poke-wasm 追加後も回帰なし)。
- 列挙器(クロスチーム限定): hp_buckets=3 で total=648=8·3⁴・valid=288=8·(h-1)²·h²、
  26 で total=3.66M・valid=3.38M=8·25²·26²(理論値一致)。JS/Rust/直接infer の3者が index・値とも一致。
- **整合スポット検証**: JS の `denseIndex` 式が Rust 列挙順と一致(k=9458 を要求→9458 返却、
  HP/種族も整合)。表は infer_encoded 由来なので値も一致。
- **ヘッドレス通し**: Node で WASM+テーブルを結合し1試合完走(37ターン決着)、pSwitch 平均0.567・
  range[0,1]、満タン Cloyster ミラーで効果抜群ダメージ 36–43% を確認。
- 静的配信スモーク(python http.server): wasm=application/wasm 等 MIME 正常、bin 14.6MB 到達。
- `uv run pytest tests/test_ckpt_tournament.py` 7 passed(既存経路に回帰なし)。
