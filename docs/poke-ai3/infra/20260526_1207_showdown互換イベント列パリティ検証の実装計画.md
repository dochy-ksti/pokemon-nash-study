# Showdown 互換イベント列パリティ検証 — 実装計画

作成: 2026-05-26 12:07 JST

## 目的とゴール

poke-sho-rust が Pokemon Showdown と「同じ出力」を生成することを保証する。
ただし **Showdown の RNG ストリームをビット再現するのではなく**、Showdown を
oracle(真実)とし、poke-sho-rust が同じ出力を再現できるかを検証する方針を取る。

- Showdown が「どの判定点で何回 `random()` を引くか」の引き順を完全一致させるのは
  脆く高コスト。微小な draw ズレでシード列がずれると以降全部が不一致になる。
- 代わりに、Showdown の出力を真実とし、各ヒットについて poke-sho-rust 側で
  (急所有無 × 16段ダメージ乱数) を総当たりして「再現できる組み合わせが存在するか」を
  検証する。これは RNG 一致ではなく **ダメージ式・機構の Showdown 互換性検証**。
- 急所は Showdown が `|-crit|` を出すので観測可能 → 急所は観測値に固定し、
  実質 **16 ロールのみ総当たり**(crit フラグは観測値との一致チェックも兼ねる)。

### 現フェーズのスコープ前提

命中率 100%・追加効果なし・1ヒットのみの技(phase1: Mew + Tackle/Strength)。
この範囲で合わせる。多段/命中判定/副次効果は将来フェーズで拡張。

## 設計判断(確定事項)

1. **手列の供給**: 外部ドライバが同一手を両者へ供給。エージェント側 RNG は検証対象外。
   実際には検証時、ランダムエージェントは Showdown のみを駆動し、poke-sho-rust は
   その出力を replay する(後述)。
2. **出力の比較**: 文字列の完全一致は不要。`ShowdownTrait` をイベント列受け渡しに変更し、
   Showdown プロトコル文字列をイベント列へ変換する。poke-sho-rust が生成するイベント列が
   Showdown 由来のイベント列と一致することがゴール。
3. **イベント語彙**: Showdown プロトコルを型付きでミラーした共通 `Event` 型。
   **poke-sho-rust に定義**(poke-env-rust は既に poke-sho-rust に依存)。必要なら後で別クレート。
4. **識別キー**: ポケモン・技は **Showdown 正規化 ID 文字列**(`toID` 相当)で統一。
5. **比較の射影**: Showdown の豊富なイベント列を共通 Event 語彙へ射影して比較し、
   語彙外(`|t:|` `|turn|` `|upkeep|` `|-supereffective|` `|-resisted|` `|gen|` `|player|` 等)は
   破棄。型相性は HP に反映されるので破棄しても検証可能。phase 進行ごとに語彙へ昇格。
6. **検証粒度**: ヒットごとに Showdown の `|-damage|` 後 HP をアンカーにして独立照合。
   計算量はヒット数に線形。
7. **検証の深さ**: エンジン全体(`apply_turn`/step)を駆動し、生成 Event 列**全体**を一致判定。
   ダメージ式に加え faint タイミング・HP クランプ・勝敗判定まで網羅。
8. **ダメージ式**: 今のうちに Showdown 完全互換へ書き換え(下記詳細)。
9. **検証ハーネス配置**: poke-env-rust の統合テスト(showdown CLI 有無で gate)、N シード。
10. **step API**: crit/roll を明示引数に取る決定的関数へリファクタ。
11. **ShowdownTrait**: `Update` を `Vec<Event>` に。`Request` は **行動の可否を表す型**(phase1 最小)。
12. **poke-sho-rust 単体の RNG**: 現行 LCG のまま(パリティと無関係)。
13. **行動順**: replay では Showdown の `|move|` 観測順を使い、`apply_turn` の order 決定に依存しない。

## Showdown 側の確認済み事実

- seed は `>start {"formatid":..,"seed":[s0,s1,s2,s3]}` で数値配列として渡しており、
  Showdown はこの形式だと **Gen5RNG**(64bit LCG, a=0x5D588B656C078965, c=0x269EC3, 上位32bit出力)を選択。
  ※ 本方針では RNG 再現はしないため、この事実は参考情報。
- `simulate-battle` は `BattleTextStream`(omniscient)を使用。`update` ブロックは
  **生ログをそのまま含み**、`|split|pN` マーカーと「secret 行(実数HP)→ public 行(パーセント)」の
  両方を出力する。→ 実数 HP は secret 行から取得可能。
- `battle.ts` はバトル開始時・各ターン境界で `|t:|<unix秒>` を出す(壁時計依存・非決定 → 射影で破棄)。
- ダメージは `getDamage` → `modifyDamage`。crit と randomizer は単純 trunc、
  STAB/型補正がある時だけ 4096 固定小数 `modify` と段階的 doubling が効く。
  phase1 の Mew(Psychic)/Normal 技は STAB なし・型補正 1倍なので現状式と既に一致するが、
  先行して一般化する。

## 実装タスク

### A. 共通 Event 型(poke-sho-rust)

- Showdown プロトコルを型付きでミラーした `Event` を定義(正規化 ID 文字列で識別)。
  phase1 で必要な variant:
  - `Move { user: PokemonRef, move_id: String, target: PokemonRef }`
  - `Crit { target: PokemonRef }`
  - `Damage { target: PokemonRef, hp: u32, max_hp: u32, fainted: bool }`(実数 HP)
  - `Faint { target: PokemonRef }`
  - `Win { player: String }`
  - `Tie`
- `PokemonRef` は `(player, slot, species/正規化名)` 程度。
- 既存の phase1 `Event`(`MoveId`/`Player` 直持ち)からの移行。
- 比較用に `PartialEq` と、必要なら正規化用のヘルパ。

### B. ダメージ式の Showdown 完全互換化(poke-sho-rust `damage.rs`)

- `tr(x)`: truncate(正値は floor 相当)。
- `modify(v, num, den) = tr((tr(v * tr(num*4096/den)) + 2047) / 4096)`。
- base = `tr(tr(tr(tr(2*level/5+2) * power * atk) / def) / 50)`、その後 `+2`。
- crit = `tr(base * 1.5)`(`modify` ではない)。
- randomizer = `tr(base * (100 - roll) / 100)`、`roll = random(16)` ∈ 0..15(=85..100%)。
- STAB = `modify(base, 1.5)`(4096 固定小数)。
- 型相性 = `typeMod > 0` は段階的 `base *= 2`、`typeMod < 0` は段階的 `base = tr(base/2)`、
  免疫は 0。`clampIntRange(typeMod, -6, 6)`。
- 既存テストの期待値を Showdown 値に合わせて再確認。

### C. step API(poke-sho-rust `phase1.rs`)

- `execute_action` を `(state, attacker, move, crit: bool, roll: u8) -> (Vec<Event>, BattleState)` の
  **決定的関数**へリファクタ(public 化)。
- 本番(`apply_turn` 経由・LocalShowdown)は `BattleRng` から crit/roll を引いてこの関数を呼ぶ薄い wrapper。
- 既存の `BattleRng`/`LocalRng` の挙動は維持。

### D. ShowdownTrait のイベント化(poke-env-rust `showdown_trait.rs`)

- `ReceiveInfo::Update` を `Vec<Event>`(共通型)に変更。`UpdateItem` 廃止。
- `ReceiveInfo::Request` を型付きへ:
  `Request { moves: Vec<MoveOption { id: String, disabled: bool }> }`(phase1 最小、アクティブ1体)。
  交代・force_switch・team preview は phase 進行時に拡張。

### E. Showdown プロトコル → Event パーサ(poke-env-rust)

- `update` ブロックの行を共通 Event に変換。認識できない行は破棄(射影)。
- **`|split|pN` 対応**: 直後の secret 行(実数 HP)を採用し、続く public 行を破棄(現 reader 未対応)。
- 技表示名("Tackle")→ 正規化 ID("tackle")変換。ポケモン ident("p1a: Mew")の解析。
- `sideupdate` の `|request|` JSON → 型付き `Request` へパース(選択可能な技 slot のみ抽出)。

### F. LocalShowdown のイベント化(poke-env-rust `local_showdown.rs`)

- poke-sho-rust の `Event`(B/A で共通化)をそのまま `ReceiveInfo::Update(Vec<Event>)` で流す。
- `build_request_json` を型付き `Request` 生成へ置き換え。

### G. 検証ハーネス(poke-env-rust 統合テスト)

- showdown CLI が無ければ skip(既存 `run_phase1_random_battle` と同様の gate)。
- N シード × ランダム手列でループ。各ゲーム:
  1. シード付きランダムエージェントで Showdown(`SimulatorShowdown`)を1ゲーム駆動 →
     射影 Event 列 `E_sd` を収集。
  2. poke-sho-rust を初期化。
  3. `E_sd` を**観測した `|move|` 順**に行動ごとに replay:
     - 各 `Move` イベントの move を step API に渡す。
     - 観測した `|-crit|` の有無で crit を固定し、`roll ∈ 0..16` を総当たり。
     - その行動の `|-damage|` 後 HP(実数)を再現する roll を採用(存在すれば OK、複数一致可)。
     - 状態を進め、生成 Event を蓄積。
  4. poke-sho-rust の生成 Event 列が `E_sd` と完全一致するか assert。
     どのヒットでも再現 roll が無ければ **不一致として診断付き panic**
     (どのヒット・観測ダメージ・poke-sho-rust の16候補ダメージを出力)。

## 留意点 / 既知の非分岐事項

- ロール曖昧性: 複数ロールが同一 HP を再現しても、存在すれば合格(Showdown が使った
  正確なロールを復元する必要はない)。比較対象は HP/crit/faint なので無害。
- 双方向性: 射影後の Event 列の長さ・内容一致を見るので、poke-sho-rust が余分な
  イベントを出すケースも検出できる。
- 1v1 KO 時: Showdown は `|faint|` → (upkeep等を挟んで) `|win|`。射影で装飾を落とすと
  `Faint` → `Win` 順になり poke-sho-rust と一致する想定。ハーネスがズレを検出する。

## 将来フェーズへの拡張点

- 命中判定・副次効果・多段技 → 総当たり空間と Event 語彙を拡張。
- 交代・force_switch・team preview → `Request` 型と Event 語彙を拡張(phase4〜)。
- アイテム・天候・特性 → ダメージ式の `modify` 機構が既に一般化済みなので modifier を追加。

## 実装順(推奨)

1. A 共通 Event 型 + B ダメージ式互換化(poke-sho-rust 内で完結、テスト容易)。
2. C step API リファクタ。
3. D/F ShowdownTrait と LocalShowdown のイベント化。
4. E Showdown パーサ(`|split|` 対応含む)。
5. G 検証ハーネス。
