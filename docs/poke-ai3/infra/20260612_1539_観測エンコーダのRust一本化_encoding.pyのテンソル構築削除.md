# 観測エンコーダの Rust 一本化 (encoding.py のテンソル構築削除)

## 背景

観測 (`StateForPlayer`) → モデル入力テンソルのエンコードが二重実装だった:

- Rust: `poke-ai3/src/obs_encode.rs` の `encode_batch` (学習ホットパス、
  rust-numpy ゼロコピー経路、コミット bbe5ec2 で導入)
- Python: `poke-ai3-python/python/poke_ai3_train/encoding.py` の
  `encode_observations` (play_gui / inspect_matchup / supervised が使用)

観測フィールドを増やすたびに両方を同期修正する必要があり、ズレるとサイレントに
壊れる状態だった。本作業で Python 側のテンソル構築を削除し、全経路が Rust の
`encode_batch` を唯一のエンコーダとして使う形にした。レイアウト・dtype は完全互換
(既存 checkpoint はそのまま使える)。

## 現状調査の結果 (4経路の実態)

- `agent.py` (`infer_encoded`): Rust エンコード済み numpy dict を
  `EncodedObservations` (torch) に包み直すだけで、テンソル構築はしていなかった。
- `play_gui.py`: `HumanGame.observation()` の JSON を parse した
  `StateForPlayer` dict を `encode_observations([{"state": obs}], ...)` に渡す。
- `inspect_matchup.py` / `supervised.py`: trajectory item
  (`{"state": StateForPlayer dict, ...}`) の list をそのまま渡す。

→ 全 call site が「StateForPlayer の JSON 由来 dict」を持っているため、
JSON 文字列を受けて Rust でエンコードする口を 1 つ足せば全経路を賄える。

## 変更内容

### Rust

- `poke-ai3/src/obs_encode.rs`: `encode_states(&[StateForPlayer]) -> EncodedBatch`
  を追加。ルーティング情報を持たない経路用で、`game_id`/`request_id` に行番号、
  `player` に P1 をダミーとして詰めて `encode_batch` に委譲する (テンソル列には
  影響しない)。`encode_batch` とテンソル列が一致するテストも追加。
- `poke-ai3-python/src/encoded.rs`: pyo3 関数 `encode_observation_states(states_json)`
  を追加。`StateForPlayer` の JSON 配列文字列を deserialize → `encode_states` →
  既存の `encoded_batch_to_dict` で numpy dict 化。
- `poke-ai3-python/src/lib.rs` / `python/poke_ai3/__init__.py`: 上記をモジュール登録・
  エクスポート。

学習ホットパス (`recv_observations` → `recv_encoded` → `encoded_batch_to_dict`) は
無変更で、性能影響なし。

### Python

- `encoding.py`: テンソル構築ロジック (~90 行) を削除。残したのは:
  - `EncodedObservations` dataclass — モデル入力の窓口として存続
    (`model.py` の forward が受ける型、`__getitem__` で minibatch スライス)。
  - `encoded_from_arrays(arrays, device)` — Rust numpy dict → torch 変換の共通
    ヘルパ (新設。agent.py にあった包み直しをここへ集約)。
  - `encode_observations(items, device)` — シグネチャ据え置きで中身を
    `json.dumps(states)` → `encode_observation_states` → `encoded_from_arrays`
    に置換。**このため play_gui / inspect_matchup / supervised は無変更**。
- `agent.py`: `infer_encoded` の包み直しを `encoded_from_arrays` 呼び出しに変更。

### テスト

- `poke-ai3-python/tests/data/make_golden.py` + `encoding_golden.json`:
  **移行前**の純 Python 実装の出力を 2 観測 (控え空き枠あり/フル) でゴールデン化。
- `poke-ai3-python/tests/test_encoding.py`: 新経路の出力がゴールデンと
  値・shape・dtype すべて一致することの回帰テスト + 空バッチテスト。

## 検証結果

- `cargo test --workspace`: 全パス (obs_encode 3 テスト含む)。
- `uv run maturin develop` + `uv run pytest tests/`: 2 passed
  (ゴールデン一致 = 旧 Python 実装と bit 一致)。
- スモーク: play_gui / inspect_matchup / supervised / agent の import と、
  `HumanGame` の実観測 1 件の `encode_observations` が通ることを確認。

## 今後の注意点

- 観測フィールドを追加する際の修正箇所は Rust 側のみ:
  `observation.rs` (StateForPlayer) → `obs_encode.rs` (EncodedBatch) →
  `encoded.rs` (dict 化) → Python は `EncodedObservations` のフィールド追加と
  model 側の消費だけ。テンソル構築の二重管理は不要になった。
- フィールドを変えたらゴールデンは再生成が必要
  (`uv run python tests/data/make_golden.py` — ただしその時点では新経路の出力を
  固定する意味になるので、レイアウト変更の意図確認用として使う)。
- 相手控え HP トラッキング (docs/poke-ai3/stage3/20260608_2243 の次タスク) は
  この一本化の上に乗せられる。観測次元が変わるので checkpoint 非互換・再学習要
  という点は従来どおり。
