# 観測転送の JSON 廃止と Rust エンコード化 (設計判断まとめ)

作成: 2026-06-11 23:59 JST (grill-me セッションでの合意事項)

## 背景 / 動機

学習が遅い原因を調査した結果、GPU (RTX 5090) 使用率 1%・Python 1スレッド 100% という
プロファイルを確認。ボトルネックは GPU でも Rust シミュレーションでもなく、
**Python⇔Rust の JSON 境界と Python 側エンコード処理 (GIL 下で直列)** だった。

- データの流れ: Rust executor → JSON 文字列 → `json.loads` →
  `encoding.py` の Python ループでテンソル化 → モデル forward →
  `json.dumps` → Rust に返送 (`train_loop.py:116-122`)
- モデルは極小 (Embedding + pointer) なので forward は数百μs。GPU はほぼアイドル。
- ベンチ実測 (5 epoch, stage 3b, no-random/no-crit):
  | 設定 | 時間 | 備考 |
  |---|---|---|
  | A: sims 64, W 16, batch 既定(256) | 1:04 (12.8s/ep) | 従来設定 |
  | B: sims 512, W 256, batch 8192 | 6:44 (80.7s/ep) | ハンドオフ案 |
  | C: sims 512, W 256, batch 4096(既定) | 7:32 | B より遅い |
- B < C は「バッチを深くして往復回数を減らすほうが速い」= Python 律速仮説と整合。
- 結論: sims / sim_concurrency / max_batch_size の調整では頭打ち。境界自体を直す。

※ ハンドオフ (20260608_2243) の `--chunk-threshold` はリファクタリング (f446f03) で
`--max-batch-size` に改名済み。既定値 `num_games * sim_concurrency / 2` は不変。

## 決定事項

### 1. スコープ: 観測 (recv_tensors) と推論返送 (send_inferenced_data) のみ

JSON 境界は3箇所あるが、毎推論ステップで発生するホットパス2本に絞る。
`recv_trajectories` は低頻度なので JSON のまま残し、効果を実測してから判断する。

### 2. 転送形: Rust 側で最終テンソル形まで組み立てる

汎用中間バイナリではなく、`encoding.py` の `EncodedObservations` と同じ最終形の
配列群を Rust が直接構築し、rust-numpy で numpy 配列としてゼロコピー返却する。
Python 側は `torch.from_numpy` するだけになり、`encode_observations` の
Python ループがホットパスから消える。

返す配列 (B = バッチ行数):

- gid 系 (i64): `my_species (B,)`, `opp_species (B,)`, `move_gids (B, NUM_MOVES)`,
  `bench_species (B, NUM_BENCH)`, `bench_move_gids (B, NUM_BENCH, NUM_MOVES)`
- 特徴系 (f32): `my_hp (B,)`, `opp_hp (B,)`, `move_present/move_legal (B, NUM_MOVES)`,
  `bench_hp/bench_present (B, NUM_BENCH)`, `bench_move_present (B, NUM_BENCH, NUM_MOVES)`,
  `switch_legal (B, NUM_BENCH)`
- マスク (bool): `legal_action_mask (B, ACTION_DIM)`

レイアウト定数 (`NUM_MOVES` / `NUM_BENCH` / `ACTION_DIM`) は従来どおり Rust が唯一の正。

### 3. ルーティング: ステートレスにエコーバック

`recv` がテンソル群に加えて `game_id (B,)` / `player (B,)` / `request_id (B,)` の
int 配列を返し、Python は同じ行順の `policy (B, ACTION_DIM)` / `value (B,)` と
ともにそのまま `send` へ返す。Rust は行ごとに game チャネルへ振り分ける。

却下案: Rust が直近バッチのルーティングを内部保持して Python は policy/value だけ
返す案。転送量は減るが FIFO の暗黙前提が生まれ、in-flight 複数バッチ
(`prepared_observations` は deque) や評価系の複数経路で壊れやすい。

### 4. JSON 経路は等価性検証後に完全削除

- numpy 版実装後、executor の JSON ループ利用箇所3つ
  (`train_loop.py` / `eval_vs_rule.py` / `inspect_matchup.py`) を全て切り替える。
- 移行時に「同じ観測を Rust エンコードと `encoding.py` の両経路で処理して
  全テンソル一致を assert する等価性テスト」を書き、合格を確認してから
  JSON 版メソッド (`recv_tensors` / `send_inferenced_data`) と `agent.infer` の
  dict 経路を削除する (検証テストも JSON 経路ごと削除)。
- 二重実装を残すと片方だけ直すバグの温床になるため残さない。旧実装は git 履歴で参照可。
- `encoding.py` 自体は削除しない: trajectories (JSON のまま) / `play_gui.py` /
  `supervised.py` が dict ベースで使い続ける。

### 5. 実装の置き場所: poke-ai3 にエンコード本体、pyo3 側は薄いラッパ

- `poke-ai3` に新モジュール (例 `src/obs_encode.rs`):
  `Vec<PlayerObservation>` → フラット `Vec<i64>` / `Vec<f32>` 群 (形状込み構造体)。
  pyo3 なしで `cargo test -p poke-ai3` により単体テスト可能。
- `poke-ai3-python`: rust-numpy (`numpy` crate) でフラット配列を `PyArray` に包む
  バインディングのみ。Cargo 依存に `numpy` を追加。
- `poke-env-rust` には手を入れない (モデル固有のテンソルレイアウト知識を env 層に
  漏らさない)。
- ファイルは 300 行制限 (AGENTS.md) 内に収める。

### 6. 完了条件と後続の流れ

1. `cargo test --workspace` + 等価性テスト合格 → JSON 経路削除。
2. 計測 B と同条件 (sims 512, W 256, batch 8192, 5ep) でベンチ再実行し、
   高速化幅を `experiments/poke-ai3/` に記録 (計測 A/B/C もまとめて記録)。
3. 速くなったことを確認後、本番 **30 epoch** (300 ではない・ユーザー指示) を
   新チェックポイント (例 `stage3b_fast30.pt`) で学習し、`eval-vs-rule` で
   現行 `stage3b_long_xteam.pt` (0.564) と比較。

## 互換性に関する確認事項

- 今回の変更は **転送方法のみ**。観測の内容・次元は変えないため、
  既存 checkpoint (新エンコーディング d3ae347 以降) はそのまま使える。
- 数値が完全一致なら学習も決定論的に同等のはず (等価性テストで担保)。
