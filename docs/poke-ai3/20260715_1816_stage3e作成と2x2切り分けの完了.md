# Stage 3e 作成と 2×2 切り分けの完了

## このセッションでやったこと

Stage 3c の各個体に Crunch / Dark Pulse を追加した `Stage 3e` を新設し、H=26 の厳密 Nash を
解いて web 配信まで完了した。3b/3c × 等倍打撃なし/あり の 2×2 格子が完成した。

実験の詳細・全数値は
`experiments/poke-ai3/20260715_1816_Stage3e対称ゲームでの等倍打撃追加と2x2切り分け.md`
に記録した。以下は引き継ぎに必要な要点のみ。

## 主要な結果

- **2×2 の切り分けが決着した**。3c→3e の効果（交代率 54.1%→24.5%、平均ターン 31.3→11.1）は
  3b→3d（55.1%→23.6%、30.0→11.0）とほぼ同一。3d で見えた交代の激減は Stage 3b の非対称性
  ではなく**等倍打撃そのものの効果**である。
- 3e の exploitability は mean `0.00000184` / max `0.00009811`。393秒で収束。
- 完全対称ゲームなのでミラー恒等式 `V(s)+V(swap(s))=1` を全3,380,000ペアで検査でき、最大誤差は
  u16 の1刻み（0.001）に収まった。3d では不可能だった強度の検証。
- 反直観的な副産物: **どのダメージ乱数でも劣って見える技が、確率1で使われる状態がある**。
  相手が瀕死寸前だと優れた技の超過ダメージが捨てられ、交代先へ持ち越すダメージだけが効くため。
  事前のダメージ線形比較による支配予測は 85.7% の状態でしか成立しなかった。

## 注意した設計上の制約（今後 3f 等を足すとき必ず効く）

1. **技の宣言順は `Crunch` / `Dark Pulse` / `coverage` で固定**する。`poke-env-rust` の行動
   index は「習得技スロット相対」だが、`poke-ai3-python/src/nash_vi_cache.rs` の `action_code`
   は MoveId から `Crunch=0 / DarkPulse=1 / その他の技=2 / Switch=3` を決める。両者が一致する
   のはこの宣言順のときだけである。
2. **`Stage` enum に variant を足してもコンパイルは通ってしまう**。catch-all の `_` アームが
   あるためで、実際に今回 `poke-ai3/src/game_task.rs` の `species_from_index` が 3e に対して
   `Goodra-Hisui` を返すバグを埋め込みかけた（3e は 3c と同じ通常 `Goodra`）。修正済みだが、
   新ステージ追加時は `Stage3c`/`Stage3d` を grep して同種の分岐を必ず全部見ること。

## 再利用できるようにした道具

- `poke-ai3-python/scripts/verify_nash_table.py`（新規）: 方策の量子化和・控え瀕死時の交代確率・
  ミラー恒等式を検査する。`--stage 3e` のように呼ぶ。3d の既存記録を完全再現することを確認済み。
- `poke-ai3-python/scripts/run_nash_trajectory_compare.py`: `--stages A B` で任意の2ステージを
  比較できるよう一般化した。方策形式（1技=P(交代) / 3技=4行動完全方策）は npz の配列長から導出
  するのでステージ固有の分岐を持たない。既定は従来どおり `3b 3d` で後方互換。
- `poke-ai3-python/scripts/run_3e.sh`: idempotent なドライバ。solve は最大 RSS ~11GB を使うため
  2段は必ず逐次実行する。
- `run_nash_geo_h26.py` / `export_nash_geo_web.py`: `full = stage in ("3d", "3e")` で分岐。
