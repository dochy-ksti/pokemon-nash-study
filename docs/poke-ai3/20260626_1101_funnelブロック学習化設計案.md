# funnel ブロック学習化設計案

作成: 2026-06-26 11:01 JST

## 背景

`ckpt_tournament.py funnel` は `epochs-per-step` ごとに `train-loop` を subprocess として
起動し直している。`epochs-per-step=5` のような細かい snapshot 粒度では、各チャンクごとに
torch import、Agent 構築、CUDA Graph キャプチャ、Rust executor 起動、checkpoint load が入り、
GPU 使用率が周期的に 0% へ落ちる。

関連調査:

- `experiments/poke-ai3/20260626_1047_funnel学習チャンク毎subprocess再起動でGPU0%_原因と改善案.md`
- `docs/poke-ai3/20260620_2200_チェックポイント自動選抜トーナメントとABレーティング設計.md`

## 目的

snapshot 粒度と examples 等価条件は維持したまま、学習 subprocess の再起動頻度を大きく下げる。

具体的には、`epochs-per-step=5` のままでも、`train-loop` を 5 epoch ごとではなく
例: 100 epoch ごとに起動する。`train-loop --snapshot-every 5` で中間 snapshot を保存し、
funnel 側は学習ブロック終了後にそれらを epoch 順に detector へ流す。

## 非目的

- 今回は完全 1 プロセス化までは行わない。
- 学習と head-to-head 評価は同時実行しない。
- 選抜ロジック、winrate 判定、round-robin の仕様は変えない。
- `epochs-per-step` を大きくして snapshot 粒度を粗くする対策は採らない。

## 提案

### 新規 CLI

`ckpt_tournament.py funnel` に次を追加する。

```text
--train-block-epochs N
```

意味:

- 1 回の `train-loop` subprocess で追加学習する最大 epoch 数。
- `epochs-per-step` の倍数であることを推奨する。
- 未指定時は後方互換のため `epochs-per-step` と同じ挙動にしてもよい。
- 本番 A/B では `epochs-per-step=5`, `train-block-epochs=100` を第一候補にする。

バリデーション:

- `train-block-epochs >= epochs-per-step`
- `train-block-epochs % epochs-per-step == 0` でない場合はエラー、または警告して切り上げる。
- `max-added-epochs` を超えて走りすぎないよう、各ブロックの target は
  `min(current + train_block_epochs, base + max_added_epochs)` に丸める。

### train-loop 呼び出し

現状:

```text
train-loop --max-epochs <epoch> --checkpoint-path <tag>_work.pt
```

変更後:

```text
train-loop \
  --max-epochs <block_target> \
  --checkpoint-path <tag>_work.pt \
  --snapshot-every <epochs_per_step>
```

`train-loop` 側には既に `--snapshot-every` があり、`checkpoint_path=A_work.pt` なら
`A_work_ep55.pt` のようなファイルを保存する。これを funnel 用 snapshot として使う。

## 処理フロー

### 新規開始

1. `work = data/poke-ai3/tournament/<tag>_work.pt` を準備する。
2. `epoch = base` から開始する。
3. `block_target = min(epoch + train_block_epochs, base + max_added_epochs)` を決める。
4. `train-loop --max-epochs block_target --snapshot-every epochs_per_step` を 1 回起動する。
5. ブロック内で生成された `work_ep*.pt` を列挙し、`epoch < snap_epoch <= block_target` のものを
   epoch 昇順に処理する。
6. 各 snapshot について:
   - `epoch = snap_epoch` に更新する。
   - `epoch <= warmup_until` なら head-to-head を省略し、必要なら snapshot を削除する。
   - warmup 後なら `det.feed(snapshot)` に渡す。
   - peak / finalist / state 保存は現行ロジックと同じ。
7. `finalists_target` 到達、または `max_added_epochs` 到達まで 3-6 を繰り返す。

### snapshot 名

`train-loop` が生成する名前は `<work_stem>_ep<epoch>.pt` になる。

例:

```text
A_work.pt
A_work_ep55.pt
A_work_ep60.pt
...
```

detector に渡す path はこのままでよい。既存の `A_ep55.pt` 形式へ copy する必要はない。
ただし、最終 JSON やログの読みやすさを重視するなら、処理時に `A_ep55.pt` へ rename してもよい。

推奨は **copy せずそのまま使う**。理由:

- 余計な I/O が増えない。
- `train-loop` の snapshot 出力と対応が明確。
- `discard()` は `TDIR` 配下の snapshot を unlink するだけなので、そのまま対応できる。

## resume 設計

state JSON の `epoch` は「funnel が処理済みの最終 snapshot epoch」として扱う。

resume 時:

1. state JSON を読む。
2. `work_ep*.pt` を列挙する。
3. `state["epoch"] < snap_epoch` の未処理 snapshot が残っていれば、まずそれを epoch 順に処理する。
4. 未処理 snapshot がなければ次の学習ブロックを起動する。

これにより、以下の中断に耐えられる。

- 学習ブロックは完了したが、snapshot 後処理の途中で停止した。
- warmup snapshot の削除前に停止した。
- peak 検出後、state 保存済みだが余剰 snapshot が残っている。

注意:

- `finalists` に入っている snapshot は削除しない。
- `det.history` と `peaks` に入っている snapshot も削除しない。
- 未処理 snapshot のうち `epoch > base + max_added_epochs` は処理対象外として削除候補にする。

## 余剰 snapshot の扱い

ブロック学習では、ブロック途中の snapshot で `finalists_target` に到達しても、
そのブロックの残り snapshot が既に生成済みになる。

方針:

- `finalists_target` 到達後は、それ以降の未処理 snapshot を評価せず削除する。
- `work.pt` は最新 epoch まで進んだままでよい。funnel の訓練方針はもともと
  「重みは常に最新から前進し、選抜は評価プールだけを操作する」ため。
- 過走りを小さくしたい場合は `train-block-epochs` を小さくする。

推奨初期値:

- `epochs-per-step=5` なら `train-block-epochs=50` または `100`
- まず `50` で挙動確認し、問題なければ `100` に上げる。

## ログ出力

現状の `run()` は `capture_output=True` なので、長い学習ブロック中にログが見えない。
ブロック学習化と同時に、`train-loop` 実行だけは stdout/stderr を親プロセスへ逐次流す。

最小案:

- `run_streaming(cmd)` を追加する。
- `subprocess.Popen(cmd, stdout=None, stderr=None)` または明示的な pipe forwarding を使う。
- 失敗時は return code とコマンドを出して停止する。

この変更は性能改善ではないが、長いブロックを安心して監視するために必要。

## 期待効果

例: `epochs-per-step=5`, `train-block-epochs=100`

- 旧: 5 epoch ごとに `train-loop` 起動
- 新: 100 epoch ごとに `train-loop` 起動
- 再起動回数は 1/20
- snapshot 粒度は 5 epoch のまま
- examples 等価条件は維持
- torch import / CUDA Graph capture / executor 起動の固定費を大幅に削減

## リスク

### 過走り

finalist がブロック途中で揃っても、ブロック末尾まで学習済みになる。

影響:

- A/B の選抜結果には基本的に影響しない。
- 壁時計は少し無駄になる可能性がある。

対策:

- 初期値は `train-block-epochs=50` から始める。
- 完全に止めたい場合は将来の `TrainingSession` 化で `run_until_next_snapshot()` を実装する。

### checkpoint I/O

`snapshot-every=5` で 100 epoch ブロックなら 20 個の checkpoint を保存する。

影響:

- 既存方式でも 5 epoch ごとに copy していたため、I/O 量は大きくは増えない。
- ただし train-loop が保存する snapshot と detector が残す snapshot が同じファイルになるため、
  削除ルールは慎重にする。

対策:

- `discard()` は `TDIR` 配下のみ削除する現行方針を維持する。
- state に載っている path は削除しない。

### GPU メモリ

学習 subprocess と評価は直列なので、完全 1 プロセス化より GPU メモリリスクは低い。

ただし、同じ Python プロセス内の head-to-head Agent cache は従来通り残る。
必要なら将来 `_AGENT_CACHE` に上限を入れる。

## 完全 1 プロセス化との関係

ブロック学習化は中間策。

最終形は `TrainingSession` を導入し、`Agent` と Rust executor を同一プロセスで保持したまま
`run_until_epoch()` / `snapshot()` を呼べる構成にすること。

ただし完全 1 プロセス化では、学習 Agent、optimizer state、eval Agent cache、CUDA Graph の
寿命管理が同じプロセスに集まる。設計変更としては中規模になるため、まずブロック学習化で
再起動頻度を桁で削減し、残る固定費が問題なら次段階として着手する。

## 実装手順案

1. `ckpt_tournament.py` に `--train-block-epochs` を追加する。
2. `train_to()` を `train_block_to()` に拡張し、`--snapshot-every epochs_per_step` を渡す。
3. `run_streaming()` を追加し、学習 subprocess のログを逐次表示する。
4. `work_ep*.pt` を epoch 番号で列挙する helper を追加する。
5. funnel loop を「学習ブロック生成」と「snapshot 消費」に分離する。
6. resume 時に未処理 snapshot を先に消費する。
7. `tests/test_ckpt_tournament.py` に、実対戦なしの snapshot 消費順・resume・余剰削除の単体テストを追加する。

## 推奨

最初の実装では `train-block-epochs=50` を既定候補にする。

理由:

- `epochs-per-step=5` なら再起動回数を 1/10 にできる。
- 過走りは最大 45 epoch で、`100` より心理的・実験的に扱いやすい。
- 問題がなければ driver 側で `100` に上げられる。

この変更で効果が足りなければ、次に `TrainingSession` 化へ進む。
