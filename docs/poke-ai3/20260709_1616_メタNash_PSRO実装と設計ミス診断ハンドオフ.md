**オラクル役の二重化という設計ミス**が露呈して途中停止した。原因・正しい設計・
次アクションを引き継ぐ。

## このセッションで実装したもの (`9211835`, コミット済み)

`poke-ai3-python/scripts/ckpt_tournament.py` に `--meta-strategy {latest,nash}` を追加
(既定 latest = 従来の「最新 pool_size 個一様」で挙動不変)。nash の中身:

- `_solve_meta_nash(payoff)`: 対称ゼロサム行列ゲームの Nash 混合を fictitious play で近似
  (scipy 非依存)。単体検証: RPS→[⅓,⅓,⅓]、支配戦略→[1,0,0]、n=1→[1]。
- `_extend_payoff(pool, matrix, args)`: プール総当り勝率行列を**差分更新** (新しい行/列だけ
  policy-only head_to_head)。`--matrix-n-per-side` (既定256戦/ペア) で軽量化。反対称
  (M[i][j]+M[j][i]=1、対角0.5) を検証済み。
- `_nash_window(pool, sigma, eps)`: σ≥`--nash-eps`(既定0.02) の戦略だけ敵ホスト
  (σ≈0 を捨て実効敵数=Nash サポートに頭打ち)。
- `configure_enemies(..., weights=)`: σ 比の連続ブロック配分 (Hamilton 最大剰余 `_apportion`)。
  weights=None は従来の一様 round-robin を完全温存。
- psro ループ: 中心の敵を「σ 重み付き全プール」に。行列・σ を state/出力 JSON に保存 (resume 対応)。

極小スモーク (warmup/central/exploiter 各2ep, max-iters 3〜4, 行列16戦) で新規ラン・resume とも
end-to-end 動作を確認済み (成果物は掃除済み)。

driver `scripts/run_psro_pilot.sh` に env `META=latest|nash`・`SPR=self-play-ratio`(既定0.5) を追加。
**注: SPR 追加ぶんは未コミット** (git status に run_psro_pilot.sh の M が残っている)。

## 本番パイロット PSRO_nash1 (教科書形 self-play=0) と、その失敗

ユーザ判断「教科書が0なら0でよい。p4 は全て適当でベースライン不適格」に従い、
`TAG=PSRO_nash1 META=nash SPR=0 MAX_ITERS=12` で起動。設定: 中心 warmup 200ep →
以降 50ep/iter、行列256戦/ペア、exploiter-init=target。

### 観測 (iter0..5 で停止)

exploitability (= 毎 iter の exploiter 対中心 勝率) 推移:

| iter | 0 | 1 | 2 | 3 | 4 | 5 |
|---|---|---|---|---|---|---|
| wr | 0.652 | 0.927 | 0.996 | 0.992 | 0.990 | 0.982 |

- **1.0 近くへ発散** (p4 の 0.75 横ばいより悪化)。
- **σ が {exp0,exp1,exp2} に凍結**: iter2 以降 σ=[0.41,0.38,0.21,0,0,...]。積んだ後発 exploiter は
  全部 σ=0 で集団に効かない → 非 Nash の固定点。

## 診断: オラクル役の二重化 (設計ミス)

教科書 PSRO と突き合わせると、**オラクルを1種類にすべきところ2種類にした**のが根本原因。

- **教科書 PSRO**: オラクルは「メタ Nash σ 混合への best-response」1 種のみ。それを**集団に
  追加**する。解 = 集団上の σ 混合。収束判定 = 新 BR が σ 混合に勝てなくなる (0.5 へ)。
- **今の実装の誤り**: 集団を「単一中心への best-response (exploiter)」で growth させ、別に
  「中心」を持った。結果:
  1. **指標の退化**: self_play=0 の中心は「3体の凍結 exploiter への純 BR」= 極端に狭い個体。
     それへの fresh BR (exploiter) はほぼ 100% 勝つ。exploiter-vs-中心 → 1.0 は構造的必然で、
     PSRO の exploitability では**ない** (単一純戦略の脆さを測っているだけ)。
  2. **集団 growth の退化**: exploiter 同士の内部ゲームでは汎用的な初期 exp0/1/2 が狭い後発に
     勝つため σ が最初の3体に張り付き、新個体が入らない。
- つまり「中心 = σ混合への BR」までは正しいのに、**集団に積むべきは中心スナップショット自身**
  なのに exploiter を積んでいた。

## 正しい設計 (次にやること。部品は流用可)

1. 集団 Π = **中心スナップショットの列** (別 exploiter サブプロセスは廃止 → むしろ単純化)。
2. 毎 iter: 現 σ 混合への best-response を1本学習 (= 今の中心、self_play=0 で σ混合を相手に)。
   warm-start は前 iter の中心を継続でよい。学習後スナップショットを Π に追加。
3. σ = Π 同士の勝率行列のメタ Nash (`_extend_payoff` / `_solve_meta_nash` をそのまま流用)。
4. **exploitability = 新メンバー c_k を Π に加えた時の勝率行列の新しい行 × 直前の σ**、
   すなわち Σᵢ σ_prev[i]·winrate(c_k vs Π_i)。行列を測れば追加コストゼロ。0.5 へ近づけば
   double-oracle gap が閉じて収束。
5. 成果物 = Π 上の σ 混合 (探索込みで、ゲームごとに σ で1体サンプルして着手)。単一ネットでは
   なく混合戦略が Nash 解、という当初の理解どおり。
6. 較正の反映: 最終評価は探索込み (policy-only 行列で回すなら最後だけ探索込みで測り直す)。

実装は `psro()` のループ後半 (exploiter 起動 → 中心スナップショット追加 + 行列行×σ) の
入れ替えが中心。**ユーザに「この方向で修正実装してよいか」を確認してから着手すること**
(直前にその質問を投げた状態でセッションが切れている。ワークフロー規約: 明示指示まで実装しない)。
