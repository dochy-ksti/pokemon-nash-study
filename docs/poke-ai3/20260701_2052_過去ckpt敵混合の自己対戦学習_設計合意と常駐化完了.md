## パート A: 敵混合学習の設計合意(未実装 / 次の本題)

### 動機
自己対戦のみだと非推移的に強くなり、遠い過去の checkpoint に負けやすい。遠い過去にも勝てるよう、
自分自身との対戦に加え、過去の強 checkpoint とも戦って学習する。

### 確定した設計判断(grill-me 質問1〜11の結論)

1. **学習する(ゲートだけでない)**: 過去 checkpoint と戦ったゲームから勾配更新する。ただし
   学習サンプルは**現行モデル側 (P1) の trajectory のみ**。凍結敵の手は教師にしない。
   - 現状 `build_examples` は P1/P2 両側を学習に使う([supervised.py:71-88](../../poke-ai3-python/python/poke_ai3_train/supervised.py#L71-L88))
     ので、敵ゲームでは P2 側を学習から除外する改修が必要。

2. **敵認定はゲート勝率ベース**: funnel と同粒度。既定 **X=1024 戦, Y=50%**。現行モデルの勝率が
   Y% 未満の過去 snapshot を「脅威(敵)」とする。

3. **候補は間引き snapshot 履歴**を **policy-only** で対戦させる(funnel と同じ仕組み)。warmup 中は
   判定しない(単調増加期は素通り)。

4. **学習ゲームの混合**: {自己対戦} ∪ {直近K個の敵} を**均等割り**(K=1→各50%、K=2→各33%、
   K=3→各25%)。K は CLI 掃引パラメータで「何個持てば最強か」を調べる。敵リストは recency 順の
   **サイズK スライディングウィンドウ**(新しい敵が入ると最古が学習ローテから外れる)。敵が K 未満
   なら現存分で割る。warmup 中/敵ゼロ時は自己対戦100%。

5. **敵リストへの追加は1判定=最強1個**(案B): 1回のゲート判定で複数 snapshot が勝ち越したら、
   funnel の `AllHistoryPeakDetector` と同様に総当たりで最強1個だけを peak として敵列に append。

6. **ループ構造(単一常駐プロセス)**:
   ```
   [warmup: 敵なし自己対戦]
   loop:
     1. 敵列末尾K個で1ブロック学習(自己対戦+対プールを均等割り生成)
     2. ブロック終了時 snapshot
     3. ゲート判定: 現行 snapshot vs 間引き履歴を policy-only で X=1024 戦
        → beaten_by 群 → 総当たり最強1個を peak として敵列に append
     4. loop
   終了時: 全 peak を総当たり → ラン勝者を *_finalists.json に出力
   ```
   - **実装分担**: 対プール生成(P2差し替え+P1のみ学習)は学習ループ側、判定オーケストレーションは
     tournament 側(funnel 資産流用)。ただし後述の常駐化により両者は同一プロセス内。
   - **snapshot ごとに即判定**して敵反映の遅延を最小化(旧 funnel の「一気に学習→一気に判定」の
     バッチ構造は反映が遅れるので廃す)。

7. **ゲーム単位ルーティング(実装の要)**:
   - game_id は run 全体で安定スロット([async_executor.rs:82](../../poke-ai3/src/async_executor.rs#L82),
     [game_task.rs:142](../../poke-ai3/src/game_task.rs#L142) の `loop { iter += 1 }`)。
   - `game_id % (相手数+1)` で決定論的に相手割り当て。Python の infer_step が `(game_id, player)` で
     行をグループ化し対応エージェントへ forward、元行順に scatter。
   - lookahead は rollout 内で P1/P2 両方をそれぞれの player タグで推論
     ([lookahead.rs:216-217](../../poke-env-rust/src/lookahead.rs#L216-L217))し、全推論は
     `(game_id, player)` を保持([inference_client.rs:79-84](../../poke-ai3/src/inference_client.rs#L79-L84))
     するので、**探索の内部推論にも一様にルーティングが効く**。→ Rust 改修なしで敵ルーティング可能。
   - 既存の2モデル対戦 `infer_step_split`([eval_ckpt_vs_ckpt.py](../../poke-ai3-python/python/poke_ai3_train/eval_ckpt_vs_ckpt.py))は
     player でグローバル分割なので、game_id 別ルーティングへ拡張する。

8. **探索設定**: 学習者(P1)=**探索あり**(`training_pi` が学習ターゲット)、敵(P2)=**policy-only**。
   - 根拠: ゲート判定も最終評価も policy-only なのでメトリクス整合+高速。
   - `policy_only` は現状 `LookaheadConfig` の**グローバル**フィールドで、P1/P2 とも同一 config を使う
     ([game_task.rs:497 decide_step](../../poke-ai3/src/game_task.rs#L497))。→ **(ゲームスロット, player)
     別に policy_only を可変にする Rust 小改修が必要**(自己対戦=両者探索、敵ゲーム=P1探索/P2 policy-only)。

9. **メトリクス分離**: 自己対戦勝率と対敵勝率を分離してログ出力(混ぜると win_rate 指標が読めない)。

### 評価設計(K の優劣判定)
現行 A/B と同型 (直近の supervised-epochs A/B と同じ):
各 K を1手法、**手法あたり3シード独立ラン** → 各ランで全 peak 総当たり勝者1個 → 手法あたり3勝者 →
手法間 **Bradley-Terry**(既存 rate モード)。**単発シードで結論を出さない**(±5 Elo 級ノイズ注意)。

### 次セッションの実装順(推奨)
1. Rust: `policy_only` を (ゲームスロット, player) 別に可変化。
2. Python: infer_step の game_id 別ルーティング(`infer_step_split` 拡張)。
3. Python: `build_examples` を game_id で振り分け(敵ゲームは P1 のみ学習)+ 勝率分離ログ。
4. Python: 敵プール管理(peak 列末尾K、均等割り game_id 割り当て)を常駐ループに統合。
5. CLI: K 掃引パラメータ追加。
6. 評価: 3シード × 各K → rate で BT。

## パート B: 常駐化リファクタ(実装完了)

### 目的
funnel が学習ブロックごとに `uv run train-loop` をサブプロセス起動する固定費(uv 再ビルド判定 +
torch import + Agent ロード + CUDA Graph 構築)を排除。パート A の「snapshot ごとに即判定」= 小さく
頻繁なブロックの前提でもある。eval 側は既に in-process 化済みだった。

### 変更内容
- **[train_loop.py](../../poke-ai3-python/python/poke_ai3_train/train_loop.py)**: `run_train_loop` を分解し
  **`TrainSession` クラス**新設。`__init__` が executor+Agent を1回だけ構築、
  `run_to(target_epochs, snapshot_every, checkpoint_path)` が while ループ本体を実行(到達済みは no-op)。
  `run_train_loop` は薄いラッパにして **`uv run train-loop` の挙動は不変**。
- **[ckpt_tournament.py](../../poke-ai3-python/scripts/ckpt_tournament.py)**: `train_block_to` の中身を
  サブプロセスから**モジュール常駐 `_TRAIN_SESSION` の `run_to`** に差し替え(シグネチャ維持で
  テスト stub 契約も不変)。`run_streaming`/`subprocess` import 削除。未指定スループット系は旧
  サブプロセスが依存していた train-loop argparse 既定(max_batch_size=512 / trajectories_threshold=128 /
  minibatch_size=256)を明示再現。誤記 help(minibatch「64」→「256」, train-block-epochs の
  subprocess 記述)も修正。

### 意図的な挙動差分
- **battle_seed が run 全体で固定**(executor を1回だけ構築し連続生成)。funnel は元々 `--battle-seed` を
  渡さないので運用影響なし、RNG は連続ストリームになる(むしろ自然)。

### 検証済み
- `uv run pytest tests/test_ckpt_tournament.py` → 7 passed。
- import / 等価性チェック OK。
- **小規模 funnel 実走**: 単一プロセスで 学習ブロック(ep2→3→4)↔ head-to-head eval が交互に回り、
  常駐学習 executor と ペア毎 eval executor がプロセス内共存して破綻なし。snapshot + finalists.json も
  従来形式で出力。成果物は掃除済み。
- **Rust 改修なし**(このリファクタは純 Python)。

### 未検証(次にやると良い)

  重い 3シード funnel→rate はここでは不要という結論。

### 実装レベルで確定した設計 (grill-me 第2ラウンド)
1. **役割の動的化 = 共有アトミック役割テーブル**: `Arc<Vec<AtomicU8>>`(game_id 索引)を
   executor が保持。`game_start` が iter ループ先頭で読む。Python が `set_roles` で書き換える。
   1 ゲーム 1 値 (0=自己対戦, 非0=敵ゲーム) が P2 の policy_only 上書きと trajectory タグの両方を決める。
2. **学習振り分け = Rust が trajectory に役割タグ** (`enemy_game: bool`)。build_examples はタグを見て
   敵ゲームの P2 を学習除外。ブロック境界の in-flight レースは生成時タグで吸収され correctness 保持。
3. **割当 = 剩余バケット** `bucket = game_id % (E+1)`(E=現在の敵数)。bucket0=自己対戦、
   bucket j=敵[j-1] に P2 ルーティング。均等割り自動、E=0 で全自己対戦。
4. **勝率ログ = 自己対戦 + 敵別**: 自己対戦 P1 勝率 (~50%) と対敵 P1 勝率を分離集計。
   敵ゲームは game_id 別 (w,l,d) も返し、呼び出し側が敵別に集計可能。
5. **CLI = 既存 funnel に `--enemy-window K`**。K=0 で純自己対戦 funnel を完全再現。
6. **run 構造 = 既存2段 finalist 流用 + append-only `enemy_pool`**(state 永続)。gate 検出 peak を
   毎回 enemy_pool に append(リセットなし)、学習は末尾K。run 勝者選抜は従来の
   peaks_per_rr/finalists_target 2段のまま(rate は既存で複数 finalist を BT)。
7. **推論境界 = `infer_fn` 注入**: `TrainSession.run_to` は `self.infer_fn(executor)` を呼ぶだけ
   (既定 `agent.infer_step`)。funnel が `_AGENT_CACHE` の凍結敵からプール推論 `infer_step_pool` を
   構築して注入。train_loop.py は敵ロジックを知らない。
8. **削除方針 = #4/#5 を無条件撤廃**: finalist 敗者 peak 削除と余剰掃除をやめ、run 終了まで残す
   (enemy_pool 参照と resume 再ロードのため)。ディスク主消費は #1 の非peak history 掃除で回収。
   保護ゲート不要。
