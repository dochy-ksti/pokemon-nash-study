# 人間 vs AI 対戦 GUI の設計 / Stage3b ルール対戦評価まとめ (ハンドオフ)

## 次にやりたいこと (このハンドオフの主目的)

**人間とこの学習済み AI が対戦できる HTML + JavaScript の GUI を作る。**
目的は「この AI が本当にそんなに強いのか」を人間が実際に対戦して体感・検証すること。
対象モデルは `data/poke-ai3/stage3b_long_xteam.pt` (Stage3b, クロスチーム再学習, 後述)。

## このセッションでやったこと (背景)

### 1. value 較正の大標本実測 (ハンドオフ前回タスク1)
- `inspect-matchup` を大標本 (6454 item) で回し、value は対面非依存で **+0.085 の系統的過大評価**、
  原因は target の最善手前提 (手別勝率の max)。model≈target で学習ギャップ無し。
- 過大評価は「自分中HP×相手満タん」セルに集中 (最大 +0.27)。劣勢ほどバイアス膨張。
- 記録: `experiments/poke-ai3/20260608_1641_Stage3b_value較正を大標本で実測.md`

### 2. 固定ルール最善手エージェントとの対戦評価 (ハンドオフ前回タスク2) → バグ発見
- `eval-vs-rule` を新規実装 (Rust ルール方策 + 評価コマンド)。P1=学習AI(lookahead)、
  P2=固定ルール (SE可→SE技攻撃 / SE不可→SE相方へ交代)。
- 初期結果が不可解 (有利対面で負け越し) → 検証 → **`pick_config` の同チーム対戦バグ発見**。
  Stage3b は本来クロスチーム (Team1 vs Team2) で「片側だけ SE 可 = SE可は厳密に有利」だが、
  `pick_config` は両者のチームを独立選択し **16通り中8通りが同チーム対戦** (両者SE/双方非SE の
  縮退局面) を生成、学習分布も評価も汚染していた。
- **修正**: `pick_config` を常にクロスチームに (game_task.rs)。不変条件テスト追加。
- 修正後分布で stage3b を 300ep 再学習 (`stage3b_long_xteam.pt`)。
- 再評価 (eval-vs-rule, n=512):
  | モデル | 全体 | 有利 | 不利 |
  |---|---|---|---|
  | 汚染学習 stage3b_long | 0.188 | 0.215 | 0.160 |
  | クリーン再学習 stage3b_long_xteam | **0.564** | 0.576 | 0.553 |
  | rule vs rule 基準 | 0.500 | 0.502 | 0.498 |
- 結論: 大敗はほぼ全て汚染が原因。クリーン学習で **固定ルール基準 (0.50) を上回る (0.564)**。
  学習は純粋自己対戦のみ (ルールを相手に訓練していない) で、均衡に近い混合方策が
  予測可能な固定相手を汎化的に搾取している。有利対面でちゃんと勝ち越し、直感どおりに。
- 記録: `experiments/poke-ai3/20260608_1711_...md`, `..._1759_...クロスチーム修正後の再学習300ep.md`

## ゲーム / モデルの現状 (GUI 実装に必要な前提)

### Stage3b のルール (現状の小ゲーム)
- 各プレイヤー 2 体パーティ {Cloyster, Goodra-Hisui}、先発 1 + 控え 1。
- 技は個体ごとに 1 つ。チーム編成で技が逆:
  - Team1: Cloyster=Shock Wave(電), Goodra=Bulldoze(地)
  - Team2: Cloyster=Bulldoze(地), Goodra=Shock Wave(電)
- 型相性: Shock Wave SE vs Cloyster / Bulldoze SE vs Goodra-Hisui。
- **対戦は必ず Team1 vs Team2 (クロスチーム)** → 常にどちらか一方だけが SE 可 = SE可は有利。
- 両者 Spe=90 同速。行動順は毎ターン公平なコイン (`BattleChaCha::first_player`、--no-random でも有効)。
- 交代は全技より先に解決 (`apply_turn` の `first_player` で速度順、交代優先)。
- ダメージは --no-random で最大ロール固定、--no-crit で急所なし (決定論)。
  **AI は no-random/no-crit で学習・評価しているので、GUI も当面は決定論で揃えるのが無難**
  (乱数を入れるなら学習条件とズレることに注意)。
- 行動空間 ACTION_DIM = NUM_MOVES(4) + NUM_BENCH(1) = **5** (技4 + 控え交代1)。
  技 index: Crunch=0, DarkPulse=1, ShockWave=2, Bulldoze=3。交代 index=4 (相対控え)。

### AI の推論経路 (2 段)
1. **ネット単体**: `StateForPlayer` → `encode_observations` → `model(slot_values, legal_action_mask)`
   → `(logits, value)`。logits を softmax で policy、value は盤面勝率予測 (0..1)。
   - `Agent` (poke-ai3-python/.../agent.py): `Agent(device, checkpoint_path)` でロード、
     `agent.infer(data_chunk)` がバッチ JSON を受けて policy/value を返す。
     `agent.model(...)` を直接呼ぶことも可能 (inspect_matchup の `_model_outputs` 参照)。
2. **lookahead 探索 (本来の強さ)**: `run_lookahead(oracle, state, player, seed, cfg)` が
   各合法手を rollout して `training_pi / selection_pi / value / win_rates` を返す
   (poke-env-rust/src/lookahead.rs)。rollout 中の各 ply 推論は `PolicyOracle::infer` 経由。
   - 評価/自己対戦では `InferenceClient` がバッチを Python へ送る非同期アーキ。
   - **GUI で lookahead を使うには、単一ゲーム用の同期オラクル** (torch モデルを直接呼ぶ
     `PolicyOracle` 実装) が要る。executor/バッチ機構は単一対戦には過剰。

## GUI 実装の設計案

### 全体構成 (推奨)
```
[ブラウザ HTML+JS]  <-- HTTP/JSON -->  [Python バックエンド]  <-- pyo3 -->  [Rust シミュレータ]
   盤面描画・操作                       torch モデル推論                   BattleState 進行
```
- フロント: 単一 HTML ファイル + JS (フレームワーク不要)。盤面 (両者の種族・HP・場/控え)、
  技/交代ボタン (合法手のみ活性)、ログ、そして **AI の policy 分布 / value / lookahead 手別勝率**
  を毎ターン表示する (「なぜ強いか」を見るのが目的なので AI の内部評価の可視化が肝)。
- バックエンド: `http.server` か Flask/FastAPI。`stage3b_long_xteam.pt` をロードした `Agent` を保持。

### バックエンド API 案
- `POST /new?seed=...` → 新規対戦開始。人間側 (P1) の観測 + 合法手を返す。
- `GET /state` → 現在の人間視点観測 + 合法手 + (直前の) AI の着手と内部評価。
- `POST /action {action_index}` → 人間着手を適用 → AI 着手を決定 → ターン解決 →
  発生イベント (ダメージ/瀕死/勝敗) + 新観測を返す。強制交代も同様にハンドリング。
- AI の着手は最初は **ネット greedy/サンプル**、強さ検証用に **lookahead** をオプションで。

### 必須の実装タスク (最大の難所)
**単一対戦をインタラクティブに駆動する pyo3 バインディングの新設** (poke-ai3-python/src/lib.rs)。
現状 pyo3 で公開されているのは executor (自己対戦バッチ駆動) のみ。人間対戦には以下が要る:
1. `new_game(seed, p1_team, p2_team, p1_active, p2_active)` → `BattleState` 生成
   (`BattleState::new_with_teams`, クロスチーム固定)。セッションを Rust 側で保持 or 状態を
   毎回 JSON で授受。
2. `observation(player)` → `observation_for(state, player)` を JSON 化 (StateForPlayer)。
3. `legal_actions(player)` → `state.legal_choices(player)` を ACTION_DIM index に。
4. `step(p1_choice, p2_choice, seed)` → `apply_turn` (必要なら `apply_forced_switches`) で 1 ターン
   進め、イベント列と新状態を返す。`winner()` / `is_done()` で終局判定。
   - 行動順コイン: `BattleChaCha::first_player`。強制交代サブフェーズの扱いは
     game_task.rs の `Request::ForceSwitch` ロジックを参照。
5. (任意) `ai_lookahead(state, player, cfg)`: 単一ゲーム用同期オラクルで `run_lookahead` を呼ぶ。
   - 同期オラクル = `PolicyOracle` を実装し `infer` で torch モデルを 1 件ずつ評価。
     pyo3 越しに Python コールバックを呼ぶか、候補を貯めてバッチ評価する簡易版でよい。
   - まずは lookahead 無し (ネットのみ) で GUI を通し、その後 lookahead を足すのが安全。

### 代替案 (pyo3 を増やさない手)
- Rust 側に「Human バックエンド」を足し、既存 executor の `run_player` の片側を
  「外部入力待ち」にする (game_task.rs の構造を流用)。ただし executor は無限自己対戦ループ前提で、
  人間の同期入力とインピーダンスミスマッチがあり、上の単一対戦バインディング新設の方が素直。

## 落とし穴・注意
- **決定論条件で揃える**: AI は no-random/no-crit で学習。GUI も当面ダメージ最大ロール・急所なしに。
  乱数を入れるなら「学習と評価条件のズレ」を明示し、別途検証する。
- **クロスチーム固定**: 人間が Team1 を持ったら AI は Team2 (逆も可)。同チーム対戦は縮退するので作らない。
- **行動順コイン**: 同速なので毎ターン 50/50。GUI でも seed 由来 or ランダムで公平に。AI の lookahead は
  内部で自前 rng を使う (cfg.randomize=false でもダメージは決定論だが first_player コインは引く)。
- **強制交代**: 瀕死後は交代手のみ合法。Stage3b は控え1体なので実質一択 (game_task は短絡処理)。
  人間側 UI も同様に強制交代フェーズを出す。
- **ACTION_DIM=5 (NUM_BENCH=1)**: 一部の古いメモ/探索結果に ACTION_DIM=6 とあるが誤り。
  encoding.py (NUM_BENCH=1) が正。
- **観測の隠匿**: `StateForPlayer` は相手の控え・技を隠す設計 (opp_species と量子化 HP のみ)。
  人間 UI も「相手の手の内は種族と HP%だけ見える」前提にすると AI と同条件で公平。
- **「強さの可視化」を忘れずに**: 目的は強さの確認なので、毎手 AI の policy 分布・value・
  (可能なら) lookahead 手別勝率を出すと「なぜその手か」が見えて検証価値が高い。
