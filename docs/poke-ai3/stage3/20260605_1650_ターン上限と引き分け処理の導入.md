# ターン上限と引き分け処理の導入

JST: 2026-06-05 16:50

## 目的

Stage3b で交代機構を導入した結果、「交代し続けて削らない」退化挙動により
バトルが無限に続き得る状態になった (旧 `while !state.is_done()` ループにはターン
上限がなかった)。本セッションは grill-me で設計を確定したうえで、**ターン上限と
引き分け処理を simulator 層に実装**し、`cargo test --workspace` で固めた。

## 設計 (grill-me で確定した合意)

| 項目 | 決定 | 根拠 |
|---|---|---|
| 配置層 | simulator (poke-sho-rust) | `BattleState.turn` を真実の源にし、全消費者 (local_showdown / lookahead / 学習) が自動で一貫。二重実装を避ける |
| 上限値 | 100 ターン (`MAX_TURNS` 定数) | 正常な対戦は十数ターンで決着。交代ループだけを確実に捕捉する余裕値。定数 1 か所でドリフト防止 |
| 上限到達時 | 無条件引き分け (`winner()=None`) | 最適戦略「ミラー交代→有利技」は正しく打てば有限ターンで決着。長期化=両者不適切なのでシンプルにバイアスなし |
| API 表現 | `is_done()` に上限を OR 追加、`winner` は据え置き | 最小変更で既存シグネチャ (winner: Option<Player>) を保持。引き分けは「winner=None かつ is_done」 |
| 引き分け報酬/価値 | 中立 0.5 (rollout 末端も 0.5) | 引き分けを勝ちとも負けとも見なさず、「正しく打てば勝てる」方向に勾配を向ける |

## 変更ファイル

### poke-sho-rust (simulator 本体)

- `src/battle.rs`:
  - `pub const MAX_TURNS: u32 = 100;` を追加。
  - `is_done()` に `|| self.turn >= MAX_TURNS` を追加。
  - `is_draw()` (終局かつ勝者なし) ヘルパを新設。
  - `winner()` は無変更 (引き分け = `None` のまま)。
- `src/turn.rs`:
  - `apply_turn` のターン加算 (`new_state.turn += 1`) 直後に `is_draw()` を判定し、
    上限到達なら `Event::Tie` を push して早期 return。
  - 既存の冒頭 `if state.is_done()` ガードにより、上限後の再呼び出しは no-op
    (イベント空・状態不変)。

### poke-env-rust

- `src/lookahead.rs`:
  - rollout の終局判定 2 か所を `winner()` Some チェックから `is_done()` ベースに変更。
    - 勝敗: root 視点で 1.0 / 0.0
    - 引き分け (winner=None): 中立 0.5
  - 副次効果: 交代ループや上限到達による **rollout 無限ループを構造的に防止**
    (rollout の depth_cap は小さいが、開始状態が上限近傍のとき winner=None のまま
     回り続けるエッジケースを is_done が塞ぐ)。

### 変更不要だった消費側 (確認済み)

- `poke-ai3/src/game_task.rs`: `Event::Tie` を既に処理し `winner: None` を返す
  (game_task.rs:288)。
- `poke-ai3-python/.../supervised.py`: `winner is None` を `draws` として集計済み。
- `local_showdown.rs`: ループは `while !state.is_done()`。引き分け終局時も apply_turn が
  返す `Event::Tie` を両プレイヤーへ送信してからループを抜けるため追加変更不要。

## 残課題 (Stage3b 次増分: env / model 層) — 前ハンドオフから継続

1. **poke-env-rust**: 観測に控えスロット (種族・HP・場 index、相手控えは隠匿) を追加。
   `local_showdown` の force-switch リクエスト対応。`lookahead` rollout の交代手対応
   (現状は技のみ抽出で素通し)。
2. **poke-ai3 / poke-ai3-python**: 行動空間を `NUM_MOVES + (MAX_PARTY-1)` に拡張。
   観測エンコーディングに控えスロット追加。policy_head 出力次元・fallback policy を交代込みに。
3. **学習・評価**: 両陣営ランダム先発で対戦、不利対面での適正交代率 >90% を定量判定。
   トラジェクトリ定性レビュー (battle-review) も併用。
4. **Showdown slot 整合**: Stage3b 各個体 1 技だが `MoveId::showdown_slot` は index 由来。
   env の parity 経路で slot 対応を見直す。

## 次に着手する人へ

引き分け基盤は simulator 層で固まり、学習・評価が回る前提条件は整った。
`MAX_TURNS` は battle.rs の定数 1 か所。3v3 + スピード段階で正常戦が伸びた際は
ここを測り直す (現状 100 は 1v1〜2 体パーティに十分な余裕)。次は前ハンドオフどおり
poke-env-rust の観測・request 対応から進めるのが自然。
