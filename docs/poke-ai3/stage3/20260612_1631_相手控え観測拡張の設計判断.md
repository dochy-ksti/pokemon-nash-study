# 相手控え観測拡張の設計判断

日時: 2026-06-12 16:31 JST
状態: 実装完了・A/B 比較実施済み (2026-06-12、新観測 0.635 vs 旧観測相当 0.559。
詳細は experiments/poke-ai3/stage3/20260612_1701_相手控え観測拡張のAB比較_300ep.md)

## 背景と動機

現状の観測 (`poke-env-rust/src/observation.rs` `StateForPlayer`) では、相手側は
active の種族 + 量子化 HP% しか見えない。相手の控えは種族・HP とも完全に隠匿で、
モデルは1ターンのスナップショットを forward するだけで履歴記憶もない。
このため「削れた控えへの詰め筋」「交代読み」が原理的に学習できない。

→ 相手控えの種族・HP・技を観測に入れて AI を賢くする。

前提となった直近の整備: 観測エンコーダの二重実装 (Rust `obs_encode.rs` /
Python `encoding.py`) はコミット 1338bc8 で Rust `encode_batch` に一本化済み。
今回の修正点は **Rust 2ファイル (observation.rs / obs_encode.rs) + model.py +
encoding.py (dataclass フィールド追加)**。~~encoding.py は自動追従~~ は誤り
(判断 12 参照)。

## 設計判断

### 1. 情報ポリシー: 神視点だが revealed-only に絞れるフォーマット (C案)

- 今はローカル真状態から相手控え全員を常に観測に入れる (神視点)。
- ただしフォーマット自体は Showdown 忠実 (revealed-only) と同型にしておく:
  スロットは `Option<BenchSlot>` で、`None` の意味を将来「未公開」に広げる
  だけで B (Showdown 忠実) に移行できる。瀕死は `hp_frac == 0` と恒等。
- 理由: 現フェーズ (Stage3b、チーム構成固定で実質既知) では team preview 相当で
  種族既知とみなせる。revealed-only の履歴トラッキング (「最後に見えた HP%」の
  保持、observation_for の非純関数化) は重く、ダメージ推測などと一緒に別フェーズで
  やるべき。

### 2. 観測フィールド: BenchSlot 再利用、技も神視点で見せる

- 相手控えは自軍用 `BenchSlot {species_gid, hp_frac, move_gids}` をそのまま再利用。
  技も隠さず真の習得技を詰める (専用の隠匿型は作らない)。
- 後で技を隠す場合は gid を 0 (未知) に潰すだけでフォーマット不変。
- `StateForPlayer` に追加: `opp_move_gids` (相手 active の技) と `opp_bench`
  (`Vec<Option<BenchSlot>>`、長さ NUM_BENCH、相手 active 除く相対 index 順)。
- 相手 active の技を入れる理由: 控えだけ技が見えて目の前の相手の技が見えないのは
  逆転した観測になるため。神視点方針なら active も含めて一貫させる。

### 3. HP 量子化: 相手側はすべて既存の opp_quantized_hp_frac

相手 active と同じ Showdown 流整数% (1%未満切り上げ・満タン未満 99% キャップ) を
相手控えにも適用。自軍は従来どおり実数比。

### 4. モデル: 相手 active 技は混ぜ込み、相手控えは末尾トークン追加 (A案)

- 相手 active の技: 独立トークンにせず、技 Embedding の present 加重平均を
  `opp_tok` に加算 (自控えの `bench_move_mix` と同じ流儀)。相手の技は行動では
  ないので pointer 対象トークンにする必然がない。
- 相手控え: 自控えと同構成のトークンを **列の末尾** に NUM_BENCH 個追加。
  末尾追加なら既存スライス (技 `[3:3+NUM_MOVES]`、自控え) がずれず
  policy ヘッド無変更。
- `NUM_TOKENS = 3 + NUM_MOVES + 2*NUM_BENCH` (3 = CLS + 自 active + 相手 active)。
  位置埋め込みは `default_model_config` が追従。

### 5. HP スカラ注入の整理 (冗長性の解消)

現状、自控えトークンは HP fraction が `mon_feat(hp)` と `bench_feat([hp, legal])` の
2経路で二重に入っている。checkpoint 非互換の作り直しになるこの機会に整理する:

- HP は全ポケモントークン共通で `mon_feat(hp_frac)` の **1回だけ**
  (my_active / opp_active / 自控え / 相手控えで対称)。
- 自控えの `bench_feat` は `Linear(1)` ([switch_legal] のみ) に縮小。
- 相手控えは追加スカラ射影なし (`_mon_token` + 技 mix のみ。legal が無いので
  射影すべきスカラが残らない)。

### 6. 技集約は平均のまま (和への変更は別実験)

和は平均より情報が多い (技数がノルムに乗る) が、
(1) 失われるのは実質技数のみで、本設定では技構成が種族にほぼ紐づくため
種族 Embedding が代替できる、
(2) トークンは加算合成 + LayerNorm のため、和だと技数に応じてトークン内の
種族・HP 成分の相対重みが変わる副作用がある、
(3) 今回の主検証は新旧観測の A/B 比較であり、集約方式の同時変更は交絡になる。
→ 今回は平均で揃え、和 vs 平均は観測拡張の検証後に独立小実験 (30ep×複数 seed) で。

### 7. 転送レイアウト (EncodedBatch)

`opp_move_gids`/`opp_move_present` (B, NUM_MOVES)、
`opp_bench_species`/`opp_bench_hp`/`opp_bench_present` (B, NUM_BENCH)、
`opp_bench_move_gids`/`opp_bench_move_present` (B, NUM_BENCH, NUM_MOVES) を追加。
レイアウトの正は従来どおり Rust。一本化時の golden JSON は新レイアウトで再生成。

## 追加の合意事項 (レビューでの詰め)

### 8. A/B のベースラインは「新コード + 相手観測マスク」

旧コミットをベースラインにすると判断 5 (HP 注入整理) が交絡するため、
ベースラインは新コードで相手側観測をマスクしたものとする。
旧コミットとの比較は参考値扱い。

### 9. マスクは Python 側で present ゼロ化 (b案)

学習スクリプトのフラグ (例: `--mask-opp-obs`) で `opp_move_present` /
`opp_bench_present` を全 0 に潰す。Rust は無変更。マスクした相手控え枠は
「位置埋め込みだけの定数トークン」として列に残るが、情報量ゼロなので
アーキテクチャ完全同一の A/B として許容する。

### 10. 敵味方の区別は位置埋め込みのみに任せる

相手控えトークンに side embedding は足さない。my/opp active が位置のみで
区別できている実績があるため。もし学習が混乱する兆候が出たら第一容疑者
としてここを疑う (実験記録に明記する)。

### 11. 検証は新旧 1 ランずつの軽量比較に縮小

6 ラン (3 seed × 2 アーム) は重すぎるため、新旧 1 ランずつとする。
1 seed では差がノイズか効果か区別できないので、結果は「スモーク +
方向感の確認」として解釈する。value 較正改善の仮説 (検証計画 3) は
ユーザーは懐疑的であり、軽く調べる程度に格下げ。

### 12. 修正範囲の訂正: encoding.py にもフィールド追加が必要

`EncodedObservations` はフィールド明示列挙の dataclass のため、判断 7 の
新フィールド 7 個は encoding.py にも宣言追加が必要 (「自動追従」は誤り)。
追加し忘れると `encoded_from_arrays` がサイレントに無視する (present 全 0
でも学習は回り、スモークで気づけない)。対策として、Rust が返したキーの
うちフィールド名・既知のルーティング系キー以外があれば警告/assert を入れる
か、golden JSON テストで新フィールドの値まで検証する。

### 13. 相手控えの並びは相対 index 順のままで良い

相手交代のたびにトークン位置の中身が入れ替わるが、スナップショット
forward のみのモデルでは実害がない。並びの再考は履歴導入・revealed-only
化 (判断 1 の B 移行) のフェーズで行う。

## 影響と注意点

- **観測次元が変わるため既存 checkpoint は非互換。再学習が必要。**
  play_gui / eval 系ツールは再学習済みモデルができるまで旧コミットでしか動かない。
- Stage3a (1v1) は相手控えが存在しないため、この拡張の効果検証対象外。

## 検証計画

1. 30ep スモーク (学習が回ること・速度劣化がないことの確認)。
2. Stage3b で新旧観測の A/B 比較: 同一条件・複数 seed・300ep 級。
   - 主指標: 対固定ルール最善手の勝率 (`eval_vs_rule`)、win_rate 曲線。
   - 副指標: 交代率。
3. 仮説を実験記録に先に書く: 相手控えの HP/種族が見えることで詰め筋・交代読みが
   可能になり、特に value の較正 (20260608_1641 で実測した歪み) が改善するはず。
