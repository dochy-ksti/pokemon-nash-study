## 目的・仮説

自己対戦は非推移的に強くなり「近い過去に勝てるが遠い過去に負ける」。funnel の gate で
検出した過去 peak を append-only の敵列に貯め、**直近 K 個を学習の対戦相手に混ぜる**
(`--enemy-window K`)。剩余バケット `game_id%(E+1)` で自己対戦と敵を均等割り、敵側 P2 は
policy-only。**「敵を何個保持すれば最強か」= 最適 K** を判定する。

仮説: K>=1 は K=0 (純自己対戦) より遠い過去への頑健性が増して手法平均レートが上がる。
ただし K を増やしすぎると自己対戦比率が下がり (K=3→自己対戦25%) 学習が薄まる可能性。
どこかに最適 K がある。

## 手法 (4 手法 = 新規3ラン + baseline流用)

house methodology 準拠: **1 手法 = 1 funnel ラン (finalists-target=3 → finalist 3個)**。
その 3 finalist を手法内サンプルとして rate プールに入れる (RC64/E5 と同型)。差が境界的
(|Δ|≲信頼帯) なら再現ラン追加。当初案の「3シード×12ラン」は旧・単段設計の名残で、
最終合意の2段構造 (option B) では 1 ラン=3 finalist で足りるため撤回。

| 手法 | --enemy-window | 自己対戦比率 (敵が満杯時) | 実行 |
|---|---:|---|---|
| K0 (baseline) | 0 | 100% (純自己対戦=従来 funnel) | **RC64 流用** (新規不要) |
| K1 | 1 | 50% | 新規 |
| K2 | 2 | 33% | 新規 |
| K3 | 3 | 25% | 新規 |

- K0 = [RC64](20260626_1615_学習教師sims_32vs64_random+crit環境_強さAB.md) 流用。
  RC64_finalists.json 確認済み: depth-skew2.0 / start=shared_init / warmup_until=50
  (warmup10×eps5) / finalists=RC64_ep110/170/295。本掃引パラメータと完全一致。
- K1/K2/K3 は RC64 と `--enemy-window` 以外すべて同一。shared_init 共有。
- K=0 は enemy コードが全て no-op なので RC64 と挙動一致 (流用の妥当性 OK)。

## 各 funnel ランのパラメータ (確立済み baseline に一致させ K だけ変える)

```bash
cd poke-ai3-python && make build && \
uv run python scripts/ckpt_tournament.py funnel --tag <TAG> \
  --start "$TDIR/shared_init.pt" \
  --enemy-window <0|1|2|3> \
  --depth-skew 2.0 --search-turn-min 4 --search-turn-max 8 \
  --sims 64 --sim-concurrency 16 \
  --train-num-games 64 --train-max-batch-size 512 --train-trajectories-threshold 128 \
  --train-minibatch-size 256 \
  --epochs-per-step 5 --train-block-epochs 50 --warmup-steps 10 \
  --peaks-per-rr 3 --finalists-target 3 --max-added-epochs 1000 \
  --n-per-side 512 --num-games 256 --stage 3b --random --crit
```

- `TDIR=/home/dochy/pokemon_ai_proj/data/poke-ai3/tournament`。
- tag 例: `K0s1 K0s2 K0s3 K1s1 ...` (手法+シード)。
- `--train-block-epochs 50`: baseline と同じブロック粒度 (gate は 5ep 刻みの各 snapshot で判定)。
  敵反映の即時性を上げたいなら block=eps5 も選択肢だが、baseline 整合を優先し 50 に据える。
- max-batch-size 512: in-flight 上限 = 64*16*2 = 2048 の 1/4 で安全 (AGENTS.md 2/3 目安内)。
- **注意**: max-added-epochs 1000 まで循環が起きないと finalist が揃わず打ち切り。K>=1 では
  敵混合で学習軌跡が変わるため循環タイミングは baseline と別。打ち切り時は warmup 短縮や
  max-added 拡大を検討。

## 評価 (全ラン完了後 rate 1回)

```bash
uv run python scripts/ckpt_tournament.py rate \
  --funnel-json $TDIR/K0*_finalists.json $TDIR/K1*_finalists.json \
                $TDIR/K2*_finalists.json $TDIR/K3*_finalists.json \
  --n-per-side 512 --num-games 256 --stage 3b --random --crit
```

- 全手法の全 finalist を 1 プールで総当たり → Bradley-Terry → **手法ごと平均レート**を比較。
- 判定: 手法平均差が信頼帯 (|Δ|≳8〜10 Elo) を超える K を「勝ち」。ノイズ帯内なら差なし。

## 実行方針

- 新規3本 (K1→K2→K3) を **1 つずつ逐次** バックグラウンド実行 (AGENTS.md: 実験は一度に1つ)。
- 各ラン ~30–60分 (E5 実績 wall≈1951s)、逐次で ~1.5–2時間見込み。各ラン >5分なので
  開始前にコマンドを提示する。
- checkpoint 掃除: 各ラン終了後、finalist 以外は残してよいが最終 finalist は保持。rate に使う
  全 finalist (RC64 含む) は rate 完了まで保持。
