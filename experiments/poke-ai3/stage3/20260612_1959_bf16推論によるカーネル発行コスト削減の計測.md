# bf16 推論によるカーネル発行コスト削減の計測

## 背景・目的

train-loop 実行中、Python メインスレッド (推論ポンプ) が単独で CPU 100% に張り付き
全体のボトルネックになっていた。py-spy `dump --native` でスタックを取ると、GPU 同期
待ちではなく **CUDA カーネル発行 (launch) の CPU コスト**が支配的で、autocast の
`cached_cast` (float32→bf16 キャストカーネル) の発行中だった。モデルが極小
(hidden 128, 2 層) なため GPU 計算より発行 CPU が律速の launch-bound 状態。

対策候補のうち最小変更の「autocast をやめて推論専用 bf16 モデルを持つ」
(`--infer-bf16`、デフォルト有効、`--no-infer-bf16` で従来経路) を実装し効果を計測した。

## 実装

- `agent.py`: `Agent(infer_bf16=True)` で学習モデルの bf16 deepcopy を保持。
  `infer_encoded` は autocast なしで bf16 モデル + bf16 入力で推論。`learn()` 後に
  `load_state_dict` で重みを同期。学習経路は従来どおり float32 + autocast。
- `train_loop.py`: `--infer-bf16 / --no-infer-bf16` フラグ追加。

## 計測 1: マイクロベンチ (固定バッチ 256、forward+probs+cpu 転送、300 iter)

```
max |policy diff|: 0.00123
autocast fp32: wall 2.405 ms/iter, main-thread cpu 2.405 ms/iter
bf16 model   : wall 2.091 ms/iter, main-thread cpu 2.091 ms/iter
autocast fp32 (2nd): wall 2.456 ms/iter, main-thread cpu 2.456 ms/iter
```

- **約 13–15% の短縮**。wall ≒ main-thread cpu であり、推論 1 回が丸ごと CPU
  (カーネル発行) で律速されていることも裏付けられた。
- policy の最大差は 1.2e-3 で実用上問題なし。

## 計測 2: 学習 10 エポックの A/B (参考値・比較不能)

```
uv run train-loop --num-games 32 --sim-concurrency 16 --sims 64 \
  --search-turn-min 6 --search-turn-max 12 --no-random --no-crit --stage 3b \
  --max-epochs 10 --battle-seed 20260612 [--no-infer-bf16 | --infer-bf16]
```

| 経路 | wall | CPU% | examples/epoch |
|---|---|---|---|
| autocast (baseline) | 38.1s | 302% | 97–129 |
| bf16 | 48.6s | 326% | 130–184 |

bf16 の数値差 (max 1.2e-3) でも方策の挙動が変わり試合が長引いたため、エポックあたりの
仕事量が 1.4 倍前後になり wall time は直接比較できない。examples あたりで見ると
ほぼ中立〜微改善程度。

## 結論

- bf16 化は正しく動き、forward 1 回の発行コストを ~13% 削減するが、ボトルネック解消
  には不十分。2.1 ms/iter が依然すべて CPU 発行コスト。
- 本命は CUDA Graphs (`torch.compile(mode="reduce-overhead")` など) で数十 launch を
  1 回のグラフ再生にまとめること。バッチ形状の固定 (パディング) が前提になる。
