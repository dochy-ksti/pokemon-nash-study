# メインループの sleep 頻度計測 — バッチ供給が脈動している

## 目的

`--sleep-seconds` の値 (0 / 1ms) でスループットが 5〜10% 変わる理由を特定するため、
メインループ唯一の sleep 経路 (train_loop.py: trajectories も観測バッチも無いときの
`time.sleep`) の呼び出し頻度・合計時間・連続性を実測した。

## 方法

`train_loop` モジュール内の `time.sleep` だけをカウンタ付きに差し替えるプローブ
(/tmp/sleep_probe.py) で 30ep を実行。「直前の sleep から 2ms 以内に再度 sleep」を
連続 (streak) と定義し、谷 (バッチ待ちエピソード) の長さを測った。

設定: num_games 32, sim-concurrency 16, sims 64, depth 6-12, sleep_seconds=0.001,
battle_seed 20260612。

## 結果

```
wall: 82.3s
sleep calls: 25593 (311/s)
sleep total: 27.1s (32.9% of wall)
sleep streaks: n=4930 mean=5.2 median=3 p90=10 max=203
```

- 推論ラウンドは約 80k 回 (970/s)。**約 16 ラウンドに 1 回の頻度で「バッチ待ちの谷」**
  が発生し、1 つの谷で平均 5.2 回 (≈5ms)、最大 203 回 (≈200ms) 連続で眠る。
- メインスレッドは wall の 1/3 をバッチ待ちで眠っている。つまり現在の律速は
  Python 側ではなく **tokio 側のバッチ生産が脈動していること**。

## 解釈 (仮説)

供給が定常でなく波打つのは、パイプラインの同期構造から説明できる:

1. **GPU ラウンドによるロックステップ化**: 全ゲーム・全ロールアウトの推論要求は
   同じ GPU ラウンドの ack で一斉に解放される。各ゲームはほぼ同じ計算量を経て
   ほぼ同時に次の観測を出すため、観測が「波」で届く。RootTask は 256 たまるまで
   flush しないので、波の間にキューが空く瞬間 (≈5ms) ができる。
2. **谷の周期**: 16 ラウンドごとに 1 谷 = バッチ 16 枚 (≈3.8k 観測) ごとに波が
   切れる。これは num_games×W = 512 の in-flight が深さ方向 (rollout の ply 進行)
   で同期していることと整合する。
3. **max=203 (≈200ms) の長い谷**: エポック境界。trajectories_threshold=32 で全
   ゲームが終了→学習→ゲーム再起動の間は観測が来ない。この間の sleep は正当。

sleep=0 が速いのは、谷の終わり (バッチ完成の瞬間) を 1ms 粒度ではなく即座に
拾えるため (谷 4930 回 × 平均 ~1ms ≈ 5s ≈ 実測差)。

## 含意・次の検証候補

- 谷の存在自体が「メインスレッドに余力があるのに供給が途切れる」ことを意味する。
  バッチ脈動をならす方が sleep チューニングより本質的:
  - RootTask の **early-flush** (キューが空く前に部分バッチを送る /
    タイムアウト付き flush) で谷を埋める。ただし過去実験でバッチを細らせると
    GPU 効率が落ちたため、`最小サイズ + 待ちタイムアウト` の併用が良さそう。
  - `max_batch_size` を下げて波長を短くする手もあるが、バッチ縮小とのトレード。
- 計測は sleep=1ms での値。sleep=0 では谷でもポーリングが回るため CPU 表示は
  上がるが、構造は同じ。
- 現状の既定は sleep=0 (最速)。1ms に戻すと 5〜10% 遅くなる (20260612_2200 参照)。
