# pump 1ラウンド内訳計測 — infer(GPU+D2H) 75% が支配、recv(dict) は 3% で無罪

## 目的

20260613_1117 で RootTask が天井でない (encode busy 24.5% / recv idle 61%) と
判明。残る容疑はメインスレッドの推論 pump (agent.infer_step) の直列処理。
pump 1 ラウンドを recv / infer / send の 3 区間に分け、どこがラウンド時間を
食っているかを直接計測する。

## 方法

`Agent.infer_step` に perf_counter 計測を追加 (agent.py):

- recv: `executor.recv_observations()` (packed dict 構築)
- infer: `infer_encoded()` (GPU forward + softmax + `.cpu()` D2H 同期を含む。
  末尾の `.cpu().numpy()` が GPU 完了までブロックするため infer 区間に GPU 時間が入る)
- send: `executor.send_inference()` (Rust 側で行ごとの routing)

5000 ラウンドごとに `pump_timing` を print。純 Python 変更 (Rust 再ビルド不要)。

## コマンド

```bash
cd poke-ai3-python
uv run train-loop --num-games 32 --sim-concurrency 16 --sims 64 \
  --search-turn-min 6 --search-turn-max 12 --no-random --no-crit --stage 3b \
  --max-epochs 30 --battle-seed 20260612 2>&1 | grep pump_timing
```

## 結果 (75000 ラウンド時点、安定値)

```
pump_timing rounds=75000 rows=17615086 avg_rows=234.9 total_s=36.69
  recv=1.21s(3.3%)  infer=27.50s(75.0%)  send=7.97s(21.7%)
  us/round[recv=16.2  infer=366.7  send=106.3]
  us/row[recv=0.07  infer=1.56  send=0.45]
```

| 区間 | /round | /row | 占有 |
|---|---:|---:|---:|
| recv (dict 構築) | 16.2µs | 0.07µs | **3.3%** |
| infer (GPU+D2H 同期) | 366.7µs | 1.56µs | **75.0%** |
| send (行ごと routing) | 106.3µs | 0.45µs | **21.7%** |
| 合計 | 489µs | 2.08µs | 100% |

## 所見

- **recv (dict 構築) は無罪**: 16µs/round・0.07µs/row で全体の 3.3%。
  packed_batch_to_dict はゼロコピー (into_pyarray ムーブ + reshape ビュー) なので
  予想どおり軽い。GIL を握った dict 構築が天井という仮説は棄却。
- **infer が支配 (75%・367µs/round)**。ただし GPU util は 30% (20260613_1057)
  なので、この 367µs の大半は GPU 演算そのものではなく **D2H 同期待ち + カーネル
  発行 + Python オーバーヘッド**の可能性が高い。`.cpu()` がメインスレッドを
  GPU 完了までブロックし、その間 pump は次のラウンドを開始できない (完全直列)。
- **send が 21.7% (106µs/round)**。send_inference が行ごとに route_item を回す
  (async_executor.rs)。0.45µs/row の純 CPU 処理がメインスレッドで直列実行。
- per-row で見ると合計 2.08µs/row → 理論上限 ~480k rows/s。実測 ~250〜430k obs/s
  はこの近傍で、pump が throughput 律速であることと整合する。in-flight を
  増やしても (1057) 伸びなかったのは、ラウンドが太るだけで µs/row が一定の
  ため (= レイテンシ律速でなく throughput 律速) で確定。

## 結論と次の一手

pump はメインスレッド完全直列で、recv→GPU forward→D2H 待ち→send を1本で回す。
GPU 30% util なのに infer が 75% を食う = **GPU 演算ではなく直列の同期/発行
オーバーヘッドがラウンド時間の主因**。打ち手の方向:

1. **infer を GPU 演算 / D2H 同期 / 発行 に分解**して、367µs のうち実 GPU が
   何µs かを確定する (CUDA events または synchronize 挟み込み計測)。GPU が
   小さければ overlap で隠せる余地が大きい。
2. **pump のパイプライン化 (ダブルバッファ)**: ラウンド N の D2H + send を
   ラウンド N+1 の GPU forward と重ねる。GPU に 70% の余力があるので、直列
   stall を隠せれば throughput が伸びる可能性。
3. **send の routing をメインスレッドから外す / ベクトル化** (21.7%、106µs/round)。

まず (1) で infer の中身を割って、GPU 実演算が小さいことを確認してから (2) の
overlap に進むのが筋。
