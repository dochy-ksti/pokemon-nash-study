# 1epごと eval 勝率曲線 (2 seed) — 冒頭~15ep の 0.000 全敗は再現、到達水準は seed 依存 (0.74 vs 0.84)

## 目的

stage3b で rule AI 勝率がエポック間で乱高下し、単一 checkpoint 評価が信頼できないと
判明 (20260613 の一連の計測)。「学習が実際どう進むか」を 1 エポック単位で追うため、
スナップショットを毎エポック保存し、各点を rule AI 512/256 試合で評価して勝率曲線を取る。
さらに別 seed で再実行し、曲線の形と到達水準が seed でどう変わるかを見る。

## 方法

- `train-loop --snapshot-every 1` で 1 エポックごとに `ckpt_epN.pt` を保存。
- checkpoint 再開対応 (`epochs = agent.training_step`、commit ba01cd6) により ep30→40→50 と
  継続学習しつつ番号を継続。
- `eval-curve`: スナップショット群を **fresh executor** で順に rule AI (P2 固定ルール) と対戦。
  executor 使い回しは重み切替でゲームが旧重み汚染するため毎回作り直す。Agent は checkpoint の
  model_config から自動再構築。各点 num-eval-games 256 (実際は並列分 overshoot、n=256)。
- モデルは hidden 128 (既定)。

## コマンド

```bash
cd poke-ai3-python
# seed1 (20260612): ep1..30 を学習 → ep30→40→50 と継続
uv run train-loop --num-games 32 --sim-concurrency 16 --sims 64 \
  --search-turn-min 6 --search-turn-max 12 --no-random --no-crit --stage 3b \
  --max-epochs 50 --battle-seed 20260612 --checkpoint-path data/poke-ai3/ckpt_curve.pt \
  --snapshot-every 1
# seed2 (20260615): ep1..50 を一気に
uv run train-loop ... --max-epochs 50 --battle-seed 20260615 \
  --checkpoint-path data/poke-ai3/ckpt_s2.pt --snapshot-every 1
# 各スナップショットを評価
uv run eval-curve --checkpoint-glob 'data/poke-ai3/ckpt_*_ep*.pt' \
  --num-games 256 --sim-concurrency 16 --num-eval-games 256 --sims 64 \
  --search-turn-min 6 --search-turn-max 12 --no-random --no-crit --stage 3b
```

## 結果 (勝率 vs rule AI, n=256/点)

| ep | 10 | 16 | 20 | 25 | 30 | 35 | 40 | 45 | 50 |
|---:|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| **seed1** (20260612) | .000 | .000 | .242 | .473 | .789 | .730 | .711 | .754 | .719 |
| **seed2** (20260615) | .000 | .008 | .145 | .453 | .750 | .801 | .855 | .820 | .844 |

seed1 全点 (ep1-50): ep1-16=0.000 → ep17 0.086 → ep24 0.539 → ep30 0.789 (ピーク) →
ep37 0.645 (谷) → ep43-50 0.72-0.77 で平坦 (0.74 前後収束)。

seed2 全点 (ep1-50): ep1-15=0.000 → ep16 0.008 → ep21 0.344 → ep30 0.750 →
ep35 0.801 → ep37 0.848 → ep40-50 は 0.77-0.86 で安定 (0.84 前後、なお微増傾向)。

## 発見

### 1. 冒頭 ~15ep の 0.000 全敗は seed 越しに再現する (頑健な病理)

両 seed とも **ep1〜15 が厳密に 0.000** (256試合×15ep = 数千戦で 1 勝もしない)。
seed1=16ep / seed2=15ep 沈黙でほぼ一致。これはノイズや eval バグではなく、
**初期モデルが rule AI に決定論的に 100% 負ける**状態が約 15 エポック続き、その後
突然 S 字で学習が立ち上がる、という共通構造。立ち上がりの形もほぼ同じ。

- 前学習では `weakness_move_rate` が ep1 から 0.9-1.0 と高く、技選択は早期から正しい。
  にもかかわらず 100% 負ける → 敗因は技選択でなく **交代 (stage3b の本質)** の崩壊が濃厚。
  正しい技を撃てても交代管理を誤って全敗、という仮説。

### 2. 到達水準は seed で大きく違う (0.74 vs 0.84、約 0.10 差)

- seed1: ep30 で 0.79 まで上がった後 **0.74 前後で頭打ち**。
- seed2: より高く登り続け **ep35 以降 0.80-0.86 で安定**、ep50=0.844、なお微増傾向。

収束後でも seed 間で **約 0.10** の差。「h128 の実力」を 1 点や 1 run で語るのは不可能で、
**複数 seed の平均±幅**でしか比較できない。先の「単一 ep100 で h128 0.738 > h256 0.619」
が誤りだった件と完全に整合。モデルサイズ比較 (128 vs 256) は最低 3-5 seed 必要と確定。

### 3. 学習軌道は「沈黙 → 急上昇 → 頭打ち/微増」の 3 相

両 seed 共通: 約 15ep 沈黙 → 10数 ep で S 字急上昇 → 0.7-0.85 で沈静。過去 run で見えた
「±0.2 の乱高下」は、十分学習が進む (ep40+) と収まり、各 run 内では収束的になる。
run 間/シード間の差が、run 内のエポック間ノイズより支配的。

## 結論と次の一手

- **最大の謎は冒頭 ~15ep の全敗** (両 seed 共通)。ここを解明・短縮できれば学習の立ち上がりが
  速くなり、実質的な高速化・安定化につながる。交代方策の崩壊 / lookahead target / value 初期化の
  切り分け (trajectory dump、初手攻撃/交代別内訳) が次の本命。
- **評価は必ず複数 seed**。到達水準のシード分散 ~0.10 を踏まえ、サイズ比較や改善検証は
  3-5 seed の平均で行う。
- seed2 がまだ微増傾向なので、収束水準の確定にはさらなる延長 (ep50→) も選択肢。
