# max_batch_size 256/384/512 再スイープ — 差はノイズ内、U 字の右側ペナルティ消失

## 目的

sleep 頻度計測 (20260613_0943) で「16 ラウンドに 1 谷」のバッチ供給脈動が判明。
谷の周期はバッチ閾値で決まるため、max_batch_size を振って谷とスループットの
関係を見る。in-flight 全スロットは num_games×2(P1&P2)×W = 1024 で、現行既定
256 はその 1/4。旧スイープ (20260604_1321) で 512 が最遅 (266s) だったのは
tokio::join 化前 (1 手に 2 度の GPU 処理) の構造によるもので、join 化後の
再計測でもある。

## コマンド

```bash
cd poke-ai3-python
time uv run train-loop --num-games 32 --sim-concurrency 16 --sims 64 \
  --search-turn-min 6 --search-turn-max 12 --no-random --no-crit --stage 3b \
  --max-epochs 30 --max-batch-size {256|384|512} --battle-seed 20260612
```

各設定 3 回、ラウンド方式 (512→384→256 を 3 周) で交互計測。sleep=0 既定。

## 結果 (wall, 30ep)

| max_batch_size | round1 | round2 | round3 | 平均 |
|---:|---:|---:|---:|---:|
| 512 (1/2) | 79.3s | 74.4s | 66.4s | 73.4s |
| 384 (3/8) | 75.4s | 66.1s | 68.6s | 70.0s |
| 256 (1/4, 既定) | 78.0s | 84.1s | 61.7s | 74.6s |

batch_stats (代表値): 256 → avg_real 234.7 / empty_ratio 0.083、
384 → avg_real 352.6 / 0.082、512 → avg_real 470.2 / 0.082。
どの cap でも実バッチは cap 近くまで太り、empty 比率は同一 (~8%)。

## 所見

- **run-to-run ばらつき (±5〜10s) が設定間の差より大きく、3 設定は統計的に
  区別できない**。ラウンド内勝者も 384, 384, 256 と割れた。後のラウンドほど
  速い傾向 (ウォームアップ/クロック?) があり、wall の絶対値比較は要注意。
- 旧スイープの「512 = 266s で最遅」という U 字右側の崖は **消失**。
  tokio::join 化 (P1/P2 同時 lookahead) でバリア同期の構造が変わったため。
  lib.rs の既定コメント「num_games*W だとパイプラインが途切れて最遅」は
  現状では当てはまらない。
- 平均では 384 がわずかに速いが (70.0s vs 73-75s)、ノイズ内。

## 結論と対応

- 既定 (num_games*W/2 = 256) は据え置き。変更を正当化する差は無い。
- バッチ閾値のチューニングでは谷 (wall の 1/3 の sleep) は解消しない。
  脈動対策の本命は依然 RootTask の early-flush (最小サイズ + 谷検出 flush)
  だが、谷の総時間 ≈27s に対し閾値 1.5〜2 倍化でも wall が動かなかった事実は
  「谷は閾値起因ではなく生産側 (rollout CPU) の供給律速」を示唆する。
  early-flush の期待値も限定的かもしれない。
- 今後この種の比較をする場合は、同一設定 5 回以上 + ラウンド交互 + samples/s
  (steady-state) 併記が必要。

## 追記: sleep プローブによる谷の構造比較 (sleep=1ms)

各 cap で sleep 計測 (/tmp/sleep_probe.py 再作成、手法は 20260613_0943 と同一):

| cap | wall | sleep total | 谷の数 | 谷の長さ mean/median/p90/max |
|---:|---:|---:|---:|---|
| 512 | 83.9s | 41.3s (49.3%) | 558 | 70.1 / 3 / 249 / 1337 |
| 384 | 75.7s | 31.1s (41.0%) | 327 | 89.8 / 4 / 272 / 2124 |
| 256 | 84.4s | 27.7s (32.8%) | 5052 | 5.2 / 3 / 11 / 298 |

- cap を上げると「浅く頻繁な谷 (256: 5052 回 × ~5ms)」が「深く稀な谷
  (384/512: 数百回 × 100〜300ms)」に変わり、**sleep 総量はむしろ増える**
  (33% → 49%)。それでも wall が同じ。
- つまりメインスレッド (GPU ポンプ) には巨大な余力があり、sleep の量・形は
  スループットに影響しない。**律速は tokio 側の rollout 生産 (CPU)** で確定。
- 含意: early-flush などバッチ供給の平滑化は無意味。次は rollout 生産側
  (tokio ワーカー利用率、num_games/W のスケール、シミュレータ自体のコスト)
  を攻めるべき。

## 追記2: CPU 利用率の実測 (pidstat/mpstat, cap=256 実行中)

環境: Ryzen 9 5950X (16C/32T)。pidstat 5 秒間隔でプロセス・スレッド別を採取。

```
プロセス全体: 851% (32 HT の 27%。mpstat 全体 26%)
メインスレッド (推論ポンプ): 66%
tokio-rt-worker ×32: 各 20〜55% (最大 53.8%、飽和なし)
```

- **どのスレッドも 100% に達していない**。rollout 生産は CPU 計算律速ではなく、
  ワーカーは大半の時間 GPU 推論の応答待ちで idle。「生産が遅い」のではなく
  「in-flight 並列度 (1024) が GPU ラウンドのレイテンシを覆い隠すのに足りない」
  レイテンシ律速 (convoy) と判明。直前の「rollout の CPU 計算が律速」という
  解釈を更新する。
- メインスレッド 66% も飽和遠く、20260612_2200 当時の「メイン 100% 律速」から
  オフロードで解消済み。いまは両側に余力があり、同期構造だけが詰まっている。
- 次の一手: in-flight を増やして谷を埋める。num_games 64 / W 32 などへの
  スケールアップ再計測 (max_batch_size も比例増) が最有力。20260612_2200 の
  「飽和点が動いたのでスケーリング再計測」とも一致。
