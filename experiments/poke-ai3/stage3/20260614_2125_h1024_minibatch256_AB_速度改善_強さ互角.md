# h1024 minibatch_size 256 A/B (GPU 谷対策・速度 vs 強さ)

## 目的

h1024 学習で GPU 使用率が毎エポックの learn ステップで周期的に低下する
(別記録 20260614 GPU プロファイル参照: 谷 = 学習ステップ。原因は ①learn 中は
推論サービスが止まる ②minibatch64×4周の小ステップで GPU を埋めきれない)。
minibatch_size を 64→256 に上げて谷を浅くし、速度と強さの変化を A/B する。

## 前提実装

`train-loop` に `--minibatch-size` を追加 (このセッション)。SupervisedConfig.minibatch_size
を上書きする。未指定なら既定 64。checkpoint には supervised_config が記録されるが
ロード時は復元しないため、再開時も CLI 値が効く。

## コマンド

学習 (mb256。mb64 は前回の h1024 と同一設定・同一 battle-seed):

```bash
make train ARGS="--num-games 32 --sim-concurrency 16 --sims 64 \
  --search-turn-min 6 --search-turn-max 12 --no-random --no-crit --stage 3b \
  --max-epochs 100 --max-batch-size 341 --battle-seed 20260615 \
  --hidden-size 1024 --num-attention-heads 32 --minibatch-size 256 \
  --checkpoint-path ../data/poke-ai3/size/h1024_mb256.pt --snapshot-every 20"
```

強さ: anchor_sweep (n=128/side=256) と eval_ckpt_vs_ckpt 直接対戦 (両 side 各 256, 計 512)。

## 結果

### 速度 (定常 s/epoch, ep20→ep100 のスナップショット時刻差 / 80ep)

| 条件 | s/epoch | 備考 |
|---|---|---|
| h1024 mb64  | 8.04 | 既存 |
| h1024 mb256 | 7.30 | 約 9% 高速化 |

GPU 使用率分布 (mb256, 16s サンプリング, 0.2s 間隔):
median=95%, p10=67%, `<50%` のサンプル = 5/80 (6%)。谷は残るが浅く・短くなった
(learn が少数の大きいステップになり Python オーバーヘッドと GPU アイドルが減る)。

### 強さ (ep100)

anchor_sweep (2 アンカー平均, n=256/対戦, SE≈0.031):

| 条件 | vs r2_ep105 | vs r2_ep130 | 強さ指標 |
|---|---|---|---|
| h1024 mb64  | 0.523 | 0.469 | 0.496 |
| h1024 mb256 | 0.461 | 0.492 | 0.477 |

直接対戦 (mb256 視点, 両 side 合算, n=512, SE≈0.022):

- mb256 264 勝 / mb64 248 勝 → mb256 勝率 = **0.516** (0.5 から約 0.7 SE = 互角)

## 結論

- **minibatch 256 は ep100 時点で強さは mb64 と統計的に互角** (h2h 0.516、anchor 指標は
  わずかに mb64 寄りだが両者ノイズ範囲)。
- **速度は約 9% 改善し、GPU の周期的な谷も浅くなる**。
- よって minibatch 256 は「強さを損なわずに速度を得られる」変更。h1024 のような大型では
  既定を 256 にしてよい。小型 (h128) では learn の GPU 比率がもともと低く効果は限定的の
  見込み (未測)。

## 次の一手候補

- ep200 まで延長して大型の収束後の強さ (サイズ単調性) を再評価する本筋に戻る。
  その際 minibatch 256 を採用して時間短縮してよい。
- minibatch をさらに上げる (512) と谷はもっと浅くなるが、教師あり 4 周の更新回数が
  減り学習が鈍る可能性。必要なら別途。
