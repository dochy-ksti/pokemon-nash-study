# 先読み max=6 × sims 2倍 実験計画

作成: 2026-06-25 13:48 JST / ステータス: 計画(未実行)

## 1. 目的

先読み深さ上限を下げて計算量を削った分を **sims 倍増(64→128)に回し**、
「浅く広く読む」構成が「深く狭く読む」構成に対して **強さ/速度のトレードオフで勝てるか** を判定する。

具体的には:

- 現行ベース **S8**(`search-turn-max 8`, `sims 64`)に対し、
- **S6x2**(`search-turn-max 6`, `sims 128`)を新規に学習・選抜し、
- 既存の **S6**(max6, sims64)/ **S7**(max7, sims64)も交えて強さを比較、
- 同時に **s/ep(生成スピード)** を独立に計測してトレードオフを評価する。

問い: 「max を 8→6 に下げて落ちた(かもしれない)強さを、sims 2倍で取り戻せるか。
取り戻せた上で S8 より速ければ採用価値がある」。

## 2. 背景: max を下げてきた経緯

`search-turn-min 4` / `depth-skew 2.0` を固定し、`search-turn-max` のみ 8→7→6 と下げてきた。
各 max で funnel により finalists 3 個を選抜し、3手法横断の総当たり + Bradley-Terry で比較済み。

成果物(既存): `data/poke-ai3/tournament/` に
`A_finalists.json`(= S8, max8), `S7_finalists.json`(max7), `S6_finalists.json`(max6),
および各 `*_state.json`(resume 用)。

これまでの所見(要約): **強さは max8 ≳ max7 ≳ max6 と単調傾向だが差はノイズ圏内**で、
手法間差 < checkpoint 間ばらつき。すなわち「max を下げても強さはほぼ落ちない(落ちても誤差)」。
→ ならば浮いた計算量を **sims に振った方が得** かもしれない、というのが本実験の動機。

注: 過去の s/ep 比較は **debug .so 取り違えバグ(commit `1c10ec6` で修正)** および
funnel の h2h オーバーヘッド混入で歪んでおり、速度結論は信頼できない。本実験では
**修正後のクリーンな環境で s/ep を独立計測** し直す(§5)。

## 3. 比較する構成

| タグ | search-turn-min | search-turn-max | sims | depth-skew | 状態 |
|---|---|---|---|---|---|
| S8 (=A, ベース) | 4 | 8 | 64 | 2.0 | finalists 済 |
| S7 | 4 | 7 | 64 | 2.0 | finalists 済 |
| S6 | 4 | 6 | 64 | 2.0 | finalists 済 |
| **S6x2(新規)** | 4 | 6 | **128** | 2.0 | **本実験で作成** |

他条件(`--num-games`, `--no-random --no-crit --stage 3b`, 初期 `shared_init.pt`)は全タグ共通。
sims 以外を S6 と完全一致させることで、**sims 倍増の純効果**を切り出す。

## 4. 強さの測り方(現行 funnel 3v3 Bradley-Terry)

固定アンカーとの対戦では相性ゲーゆえ強さが測れないため、**全履歴比較 funnel で各構成の
finalists 3 個を選抜 → 全構成横断で総当たり → Bradley-Terry レート** の現行手法を踏襲する
([20260624_1752 の手法](20260624_1752_depth_skew_AB_全履歴比較funnel再実験.md))。

### 4.1 S6x2 の funnel(finalists 3 個を選抜)

```bash
cd poke-ai3-python
make build   # native 鮮度確保(必須)

uv run python scripts/ckpt_tournament.py funnel --tag S6x2 \
  --start data/poke-ai3/tournament/shared_init.pt \
  --peaks-per-rr 3 --finalists-target 3 --warmup-steps 10 \
  --epochs-per-step 20 --max-added-epochs 4000 \
  --n-per-side 512 --num-games 64 --stage 3b \
  --depth-skew 2.0 --search-turn-min 4 --search-turn-max 6 --sims 128
```

- `--sims 128` が唯一の変更点(funnel 内の train-loop に渡る: `ckpt_tournament.py:159`)。
- 中断時は同コマンドに `--resume` を付けて継続可(`S6x2_state.json`)。

### 4.2 4構成横断の rate(Bradley-Terry)

```bash
uv run python scripts/ckpt_tournament.py rate --n-per-side 1024 --num-games 64 --stage 3b \
  --funnel-json \
    data/poke-ai3/tournament/A_finalists.json \
    data/poke-ai3/tournament/S7_finalists.json \
    data/poke-ai3/tournament/S6_finalists.json \
    data/poke-ai3/tournament/S6x2_finalists.json
```

- 全 finalists(4構成 × 3 = 12 ckpt)で総当たり、policy-only(確率着手)、先後 1024+1024=2048 試合/ペア。
- 出力の **手法平均レート** を構成間で比較。S6x2 が S6 を上回るか(sims 効果)、S8 に並ぶ/超えるかを見る。

### 4.3 強さ判定の注意

- 3 点推定の符号だけで断じない(前回知見)。S6x2 の平均レートが S6 を**明確に**上回り、
  かつ S8 と同等以上なら「sims 倍増が有効」と判断。差がノイズ圏(±数 Elo)なら「強さは不変」とし、
  §5 のスピード差で採否を決める。

## 5. スピードの測り方(s/ep を独立計測)

funnel/rate の wall 時間は h2h 評価が混入するため **速度指標に使わない**。
代わりに **train-loop 単体の固定 epoch ベンチ** で s/ep を測る(commit `1c10ec6` 修正後の
クリーン環境。debug .so 取り違えは解消済み)。

### 5.1 計測対象

「学習ループ 1 epoch あたりの壁時計時間 = s/ep」。これが自己対戦生成+学習を含む実コスト。
構成ごとに同一 epoch 数を回し、`(終了-開始)/epochs` を比較する。

### 5.2 計測コマンド(`scripts/diag_fix.sh` と同形、各構成 45ep)

公平化のため **`--battle-seed` を固定**(プロセス間の対戦差ばらつきを排除)。

```bash
cd poke-ai3-python
make build
T=../data/poke-ai3/tournament; OUT=/tmp/bench_sims.pt; EP=45; SEED=12345

for cfg in "max8 8 64" "max6 6 64" "max6x2 6 128"; do
  set -- $cfg; name=$1; MAX=$2; SIMS=$3
  cp "$T/shared_init.pt" "$OUT"
  t0=$(date +%s)
  uv run train-loop --num-games 32 --sim-concurrency 16 --sims "$SIMS" \
    --search-turn-min 4 --search-turn-max "$MAX" --depth-skew 2.0 \
    --no-random --no-crit --stage 3b --max-epochs "$EP" \
    --battle-seed "$SEED" --checkpoint-path "$OUT" > /dev/null 2>&1
  t1=$(date +%s)
  echo "  $name (max$MAX sims$SIMS): $((t1-t0))s = $(echo "scale=3;($t1-$t0)/$EP"|bc) s/ep"
done
```

注: funnel が使う `--num-games 64` ではなく、計測安定性のため diag と同じ `--num-games 32`
`--sim-concurrency 16` を用いる(s/ep の相対比較が目的なので絶対値より構成間比が重要)。
funnel と同条件の絶対 s/ep も欲しければ `--num-games 64` 版を別途1回取る。

### 5.3 期待される読み

- max6 sims64 は max8 sims64 より**速い**はず(rollout が浅い)。
- sims 64→128 は生成 rollout 数を倍増させ**遅くする**。
- 焦点: **max6 sims128 の s/ep が max8 sims64 と比べてどうか**。
  - max6x2 ≤ max8 かつ 強さ同等以上 → **採用価値あり**(同コストで強い or 同強さで速い)。
  - max6x2 ≫ max8 → sims 倍増はコスト過大。強さ向上が伴うかで判断。

## 6. 判定マトリクス(強さ × 速度)

| | S6x2 が S8 と同等以上の強さ | S6x2 が S8 より弱い(ノイズ超) |
|---|---|---|
| **S6x2 が S8 より速い** | 採用(浅く広く読むのが有利) | 速度と強さのトレードオフ。用途次第 |
| **S6x2 が S8 より遅い** | 強さ同等なら不採用(高コスト) | 不採用 |

加えて **S6x2 vs S6**(sims のみ差)で「sims 倍増そのものの効果」を分離評価する。

## 7. 実行順序

1. `make build`(native 鮮度確保)。
2. §5 の s/ep ベンチ(3構成、各 45ep、~数分)。← 先に軽い計測で速度を押さえる。
3. §4.1 S6x2 funnel(finalists 3 個、最大 4000ep、長時間。`--resume` 可)。
4. §4.2 4構成 rate(Bradley-Terry)。
5. 結果を `experiments/poke-ai3/` に記録(強さレート表 + s/ep 表 + 判定)。
