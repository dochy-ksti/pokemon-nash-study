# depth_skew A/B を自動選抜トーナメントで比較

実施: 2026-06-22 JST

## 目的

depth_skew の A/B を、恣意性のない checkpoint 選抜で比較する。

- A: `--search-turn-min 4 --search-turn-max 8 --depth-skew 2.0`
- B: `--search-turn-min 6 --search-turn-max 12 --depth-skew 1.0`（従来設定）

両者を共通のランダム初期 `shared_init.pt` から学習し、各々 history-funnel で
二次予選通過2個を選抜 → 4 ckpt で総当たり + Bradley-Terry → 手法平均レート比較。

## 経緯と方式

- 当初の「隣接 h2h で新>旧ならピーク」方式は機能せず。学習がほぼ単調改善のため
  「旧が新に勝つ」がほぼ発生せず、ピークが立たなかった (A は1000ep で2個、B は0個)。
- 原因: 学習は「現在の自分に勝つ」よう進むため直近の自分には勝てる (recency bias)。
  さらに ep1100 検証で「20前には勝つが 40前には負ける」非推移的な循環を確認。
- 対策: 0回戦を **履歴比較方式** に変更 (最新 vs 20/40/80 epoch 前、最新が過去の
  いずれかに負けたら最も強く倒した古い ckpt を 0回戦通過)。これで funnel が成立。

## コマンド

```bash
# A / B それぞれ
uv run python scripts/ckpt_tournament.py funnel --tag {A,B} \
  --start data/poke-ai3/tournament/shared_init.pt \
  --peaks-per-primary 3 --secondaries-target 2 \
  --epochs-per-step 20 --back-offsets 20,40,80 \
  --max-added-epochs 2000 --n-per-side 512 --num-games 64 --stage 3b \
  {--depth-skew 2.0 --search-turn-min 4 --search-turn-max 8 |
   --depth-skew 1.0 --search-turn-min 6 --search-turn-max 12}
# 最終
uv run python scripts/ckpt_tournament.py rate --n-per-side 512 --num-games 64 --stage 3b \
  --funnel-json A_secondaries.json B_secondaries.json
```

head-to-head はすべて policy-only (先読みなし) / 先後 512+512=1024 試合。

## 結果

二次予選通過: A = {A_ep200, A_ep500}, B = {B_ep120, B_ep500}

Bradley-Terry レーティング (平均0):

| ckpt | 手法 | レート |
|---|---|---|
| B_ep500 | B | +16.3 |
| A_ep200 | A | -1.4 |
| A_ep500 | A | -4.4 |
| B_ep120 | B | -10.5 |

手法平均: **B = +2.9 / A = -2.9 → 判定「B が優れた手法」**

## 結論

- 勝者は **B (search 6-12 / depth_skew 1.0、従来設定)**。**A (4-8 / skew2.0) への変更は
  改善にならず、むしろ僅差で劣勢。**
- ただし差は約6 Elo でノイズ圏内。B の2個は両極 (B_ep500 最強 / B_ep120 最弱) で、
  手法間の差より checkpoint 間ばらつきが大きい。
- 別途の直接対決 (両者を 1000ep 学習し ep1000 同士) でも B 51.6% (非有意) と同方向。
  複数の測り方で一貫して「B がごく僅かに上、depth_skew=2.0 採用の根拠なし」。

## 副産物・知見

- 学習は単調改善ではなく **非推移的に循環**する (ep1100 が ep1080 に勝ち ep1060 に負ける)。
  隣接 h2h での進捗判定は recency bias を拾うため不適。履歴比較 (離れた過去との対戦) が有効。
- 真の強さ比較には固定アンカー (ルールエージェント等) との勝率がより頑健な可能性。
- 注意: tmux は SSH 切断には耐えるが Linux 本体の電源断には耐えない (本実験中に
  誤シャットダウンで B を1度やり直した)。
