# Stage 3c H=26 厳密 Nash テーブル検証

## 目的

Stage 3b と同じ幾何打ち切りゼロサム確率ゲーム (`discount=0.99`) を Stage 3c で解き、
web 配信用 policy/value テーブルとして採用できるか検証する。

## 実行コマンド

`poke-ai3-python/` から以下をバックグラウンド実行した。

```bash
setsid nohup uv run python scripts/run_nash_geo_h26.py --stage 3c \
  > /tmp/psro/nash_geo_h26_3c.log 2>&1 &
```

- H=26
- discount=0.99
- backward VI 最大3000反復
- policy evaluation / best response 最大3000反復、収束閾値 `1e-6`
- 出力: `data/poke-ai3/nash_geo/nash_geo_h26_3c.npz`

## 結果

- 所要時間: 216秒
- キャッシュ: 55,164,312 entries、約0.7GB
- valid states: 3,380,000 / 3,655,808
- exploitability: mean `0.00001`、max `0.00886`
- mixed policy (`0.02 < P(switch) < 0.98`): 1,108,687 states
- u16 value の全 valid 鏡像検証 `V(s)+V(swap(s))`:
  - 最大誤差 `0.001`
  - 平均誤差 `0.00000747`
- 満タン時の4先発組合せを等確率にした value 平均:
  - Team 1: `0.500`
  - Team 2: `0.500`

固定先発の値は Team 1 視点で Cloyster/Cloyster が `0.587`、Cloyster/Goodra が
`0.413` (先発を反転すると対応する値も反転) となった。Stage 3c の対称性は種族・技・
先発の対応を含むものであり、固定した個別先発同士が必ず `0.5` になることは要求しない。

## web 書き出しと回帰検証

```bash
uv run python scripts/export_nash_geo_web.py --stage 3c
wasm-pack build poke-wasm --target web --out-dir ../web/pkg
cargo test --workspace
```

- `web/policy_3c.bin` / `value_3c.bin` は npz とバイト単位で一致。
- meta の stage、species order、total、radix を検証済み。
- `cargo test --workspace`: 全84件成功。
- 旧 web テーブルは `/tmp/psro/web_3c_pre_nash_20260715_034418/web/` に退避。

## 判定

exploitability、鏡像整合、混合状態数、web 形式整合の全基準を満たすため採用する。
