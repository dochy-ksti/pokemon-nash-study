# 観測転送 Rust エンコード化の速度ベンチ

実施: 2026-06-12 00:22 JST

## 目的

Python⇔Rust の JSON 境界 + Python 側エンコード (GIL 直列) がボトルネックという
プロファイル結果 (GPU 1%・Python 1スレッド 100%) を受け、観測転送を
「Rust が最終テンソル形まで組み立て → rust-numpy でゼロコピー返却」に置き換えた
効果を測る (設計: docs/poke-ai3/20260611_2359)。

## 変更内容 (要約)

- `poke-ai3/src/obs_encode.rs`: `Vec<PlayerObservation>` → フラット i64/f32/bool 配列群
  (encoding.py と同一レイアウト)。empty 観測はルーティング配列 (`empty_*`) のみ別送し、
  Python が推論結果と一緒にエコーバック (ack タイミングを GPU ラウンドと同期維持)。
- `poke-ai3-python`: `recv_observations()` (numpy dict) / `send_inference()` (配列直接)。
  JSON 経路 (`recv_tensors`/`send_inferenced_data`) と agent.infer の dict 経路は削除。
- 等価性検証: 移行期限定の両系返却メソッドで stage 3a/3b の実バッチを encoding.py 出力と
  全テンソル完全一致 assert → 合格を確認後、JSON 経路ごと削除済み。
- `recv_trajectories` は低頻度のため JSON のまま。

## コマンド

旧計測 B (20260611 設計判断ドキュメント内) と同条件:

```bash
cd poke-ai3-python
uv run train-loop --num-games 32 --sim-concurrency 256 --sims 512 \
  --search-turn-min 4 --search-turn-max 8 --no-random --no-crit --stage 3b \
  --max-epochs 5 --max-batch-size 8192
```

ログ: `logs_speed_bench_rustenc.log`

## 結果

| 設定 | 経路 | 時間 (5 epoch) | s/epoch |
|---|---|---|---|
| A: sims 64, W 16, batch 256 | JSON | 1:04 | 12.8 |
| B: sims 512, W 256, batch 8192 | JSON | 6:44 | 80.7 |
| C: sims 512, W 256, batch 4096 | JSON | 7:32 | 90.4 |
| **B': sims 512, W 256, batch 8192** | **Rust エンコード** | **0:29.7** | **5.9** |

- B 比 **約 13.6 倍**、従来運用の A 比でも約 2.2 倍高速。
- batch_stats: batches=1250, avg_real=6813, empty_ratio=0.168 (バッチ密度は JSON 時と同等)。
- 最大常駐メモリ 2.3 GB。

## 結論

Python 律速仮説どおり、境界の JSON 廃止で sims 512 / W 256 の深い設定が
実用速度になった。次ステップは新チェックポイント (例 stage3b_fast30.pt) での
30 epoch 学習と eval-vs-rule での現行 stage3b_long_xteam.pt (0.564) との比較。
