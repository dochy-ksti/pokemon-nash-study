# chunk_threshold スイープ: 256 が最適 (U字カーブ)、デフォルトを num_games*W/2 へ変更

## 目的

先の計測 (20260604_1249) で empty padding の大バッチ化 (cap=512) が小モデルでは性能悪化
(266s) と分かった。しかし cap=512 は in-flight 全スロット (num_games*2*W=1024) の半分で
あり、threshold バリアが深すぎた疑いがある。chunk_threshold を 128/256/512 で振り、
スループット最適点を探る。

## コマンド

```bash
cd poke-ai3-python
# chunk_threshold を 128 / 256 / 512 で変えて計時 (他は固定)
uv run phase2-loop --num-games 32 --sim-concurrency 16 --sims 64 \
  --chunk-threshold <N> \
  --search-turn-min 4 --search-turn-max 8 --no-random --no-crit --stage 2a \
  --max-epochs 60 --checkpoint-path <ckpt>
```

- 環境: RTX 5090, CUDA。num_games=32, W=16, sims=64 固定。各ゼロ初期化・計時。
- 全スロット数 = num_games*2*W = 1024。num_games*W = 512。

## 結果

| chunk_threshold | 実 GPU バッチ平均 | 壁時計(60ep) | 収束 epoch | empty 比率 |
|----------------:|------------------:|------------:|-----------:|-----------:|
| ~21 (旧 early-flush) | 21  | 160s | ~7  | —     |
| 128             | 114 | 61s  | 11  | 0.111 |
| **256**         | **227** | **50s** | 12 | 0.112 |
| 512 (=num_games*W) | 454 | 266s | 16 | 0.13  |

(旧 early-flush は別コミット時点・別計測 20260603_2217。128/256/512 は同一コード
commit 0b3f06b で --chunk-threshold のみ変更。)

## 所見 (U 字カーブ)

- **小さすぎ (旧 ~21)**: バッチが細く Python 往復の固定オーバーヘッドが回数分効く → 遅い (160s)。
- **128**: バッチ 114。256 より往復回数が増え微増 (61s)。
- **256 (最適)**: バッチ 227。往復償却とパイプライン維持の両立点 → 最速 (50s)。
- **512**: バッチ 454。root が 512 溜まるまで待つ同期バリアが深く、GPU と rollout 生成の
  パイプラインが途切れて idle → 最遅 (266s)。

→ 最適 threshold は num_games*W/2 (= 全スロット 1024 の約 1/4) 付近。empty 比率は
どの設定も ~11% で安定。収束はどれも正常 (epoch 11〜16 で car=1.0、撃ち分け正常)。

## 結論と対応

- empty padding は threshold を適切 (≈ num_games*W/2) に選べば旧 early-flush の 3.2 倍速。
  先の「大バッチ化は遅い」(20260604_1249) は cap=512 固有の問題で、結論を更新する。
- **デフォルト chunk_threshold を num_games*sim_concurrency から num_games*sim_concurrency/2
  へ変更** (poke-ai3-python/src/lib.rs)。従来デフォルトは最遅の 512 を引いていた。

## 次の候補

- num_games/W を変えても num_games*W/2 が最適のままか (最適点の num_games 依存性)。
- 大モデルでは最適 threshold が上振れするか (GPU 計算律速になると大バッチが有利になる)。
