# W=16 で num_games 16 / 32 / 64 の収束とスループット比較 (2a-det)

## 目的

sim_concurrency=16 を固定し、num_games を 16 / 32 / 64 と増やしたときの収束 epoch と
スループットを測る。num_games がデータ量・収束・速度にどう効くかを切り分ける。
決定論 (2a-det) でノイズを排し、num_games のみの影響を見る。

## コマンド

```bash
cd poke-ai3-python
for NG in 16 32 64; do
  ckpt="../data/poke-ai3/phase2_lookahead_2a_det_ng${NG}_w16.pt"
  rm -f "$ckpt"
  time uv run phase2-loop --num-games "$NG" --sim-concurrency 16 --sims 64 \
    --search-turn-min 4 --search-turn-max 8 --no-random --no-crit --stage 2a \
    --max-epochs 60 --checkpoint-path "$ckpt"
done
```

- 環境: RTX 5090, CUDA。W=16, sims=64 固定。各 NG ゼロ初期化、逐次実行・計時。

## 結果

| num_games | examples/ep | 壁時計(60ep) | 秒/ep | ex/秒 | 収束ep (初 car=1.0) | 収束まで壁時計 |
|----------:|------------:|------------:|------:|------:|-------------------:|--------------:|
| 16 | 64  | 94s  | 1.57 | 40.9 | 24 | ~38s |
| 32 | 128 | 160s | 2.67 | 48.0 | 7  | ~19s |
| 64 | 256 | 336s | 5.60 | 45.7 | 5  | ~28s |

(収束まで壁時計 = 秒/epoch × 収束epoch の概算。examples/epoch は trajectories_threshold
= num_games × 1 試合あたり約 4 trajectory state に比例)

全 NG とも 60 epoch 完走、エラーなし。終盤の correct_action_rate は 0.92〜1.0 で
いずれも収束 (小さな揺れはサンプルノイズ)。

## 所見

- **examples/epoch は num_games に線形比例** (64 → 128 → 256)。num_games は並列度という
  より「1 訓練更新あたりのデータ量 (バッチ)」として働く。
- **収束 epoch は num_games が多いほど激減** (24 → 7 → 5)。1 更新あたりのデータが増え、
  少ない更新回数で収束する。
- ただし **1 epoch の壁時計は num_games に比例して増える** (94 → 160 → 336s)。
- **生スループット (examples/秒) は num_games=32 でピーク (48)**、64 ではわずかに低下
  (45.7) = GPU 飽和を超えてオーバーヘッドが出始める。
- **収束までの壁時計は num_games=32 が最短 (~19s)**。16 は更新あたりデータ不足で更新回数が
  かさみ遅い (~38s)、64 は 1 更新が重すぎて頭打ち (~28s)。

## 評価

W=16 × num_games=32 がこの環境 (RTX 5090) のスイートスポット。生スループットも収束までの
壁時計も最良。num_games=64 は examples/秒・収束時間ともに頭打ち〜微減で、これ以上の増加は
非効率。num_games はバッチサイズとして機能するため、sim_concurrency (GPU 飽和度) とは
役割が異なる。

## 関連

- 本ランと合わせ、この環境の推奨は W=16 × num_games=32。
