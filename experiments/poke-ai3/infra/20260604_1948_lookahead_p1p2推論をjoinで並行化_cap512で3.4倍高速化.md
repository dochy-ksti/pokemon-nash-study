# lookahead の P1/P2 推論を join! で並行化: cap=512 で 3.4 倍高速化

## 目的

`rollout` のループ内で P1/P2 の手を選ぶ際、`choose_move` が `oracle.infer().await` を
P1 → P2 と逐次 await していた。これを `tokio::join!` で同時に poll させ、両 observation を
同一バッチに乗せることでバッチ密度を上げ、threshold ゲートの穴埋め empty を減らせるか検証する。

## 変更内容 (poke-env-rust/src/lookahead.rs)

- `choose_move` を `infer_move` に分割。`infer_move` は `&state` のみ使い、推論結果
  `OracleOut` を返す。root の最初の一手 (forced, `player==root && ply==0`) は推論不要なので
  `None` を返し GPU 呼び出しを省く。
- rollout ループで P1/P2 の `infer_move` を `tokio::join!` で同時に poll。
- rng を使う `sample_move` は infer 不要なので join 後に逐次実行し、`&mut rng` の借用衝突を回避。
- 挙動 (forced 判定・サンプリング順・seed) は従来と同一。

## コマンド

```bash
uv run train-loop --num-games 32 --sim-concurrency 16 --sims 64 \
  --search-turn-min 4 --search-turn-max 8 --no-random --no-crit --stage 2a \
  --max-epochs 5 --chunk-threshold {256|512}
```

- 環境: RTX 5090, CUDA。new (join!) と baseline (逐次, git stash) を各 cap で 5 epochs × 3 回。
- baseline は変更を `git stash` して同一コードベースから再ビルドして計測。

## 結果

### バッチ統計 (5 epochs × 3 回の平均, 最終累積値)

| 構成 | 指標 | baseline | new (join!) | 差 |
|------|------|---------:|------------:|---:|
| cap=256 | avg_real | 218.4 | 230.2 | +11.8 (+5.4%) |
| cap=256 | empty_ratio | 0.147 | 0.101 | -0.046 (相対 -31%) |
| cap=512 | avg_real | 435.6 | 460.8 | +25.2 (+5.8%) |
| cap=512 | empty_ratio | 0.149 | 0.100 | -0.049 (相対 -33%) |

- 3 回の再現性は高く avg_real のばらつきは ±0.3 以内。

### wall clock (5 epochs × 3 回の平均, wheel 再ビルド込み)

| 構成 | baseline | new (join!) | 差 |
|------|---------:|------------:|---:|
| cap=256 | 8.60s | 8.47s | ほぼ同等 (-1.5%) |
| cap=512 | 27.80s | 8.16s | **約 3.4 倍速** |

- 生値: base512 = 28.93 / 27.62 / 26.84、new512 = 8.19 / 8.21 / 8.08。

## 考察

- avg_real +5〜6% / empty_ratio 約 1/3 減 は両 cap で一貫。狙いどおり P1/P2 が同一バッチに
  乗りやすくなり、threshold ゲートの穴埋め empty が減った。
- wall clock の大差は cap=512 のみで顕著。原因はバッチ充填の待ち時間と推定:
  - cap=256: threshold 256 は逐次 infer でも容易に満たせ、両者ともゲートをすぐ通過 → 同等。
  - cap=512: baseline は pending が溜まりにくく threshold 512 に届くのが遅い → 各チャンクで
    ゲートのタイムアウト待ちが発生し steady-state で累積 (27s)。join! は 512 に速く到達し
    待ちが消える (8s)。
  - 単発の --max-epochs 1 計測では baseline 512 が 8.41s と速く見えたが、これは起動バースト
    (最初の 50 チャンクは満杯) が大半で steady-state の penalty がまだ顕在化していなかったため。
    5 epochs で steady-state が支配的になり本当の差が出た。

## 結論

- 小 cap では中立、大 cap ではスループット 3 倍超。`join!` 化により大きい cap を取る選択肢が
  現実的になった。baseline は大 cap を実質使えない状態にしていた。
- 明確に入れるべき変更。今後 chunk_threshold スイープ (前回 256 が最適) を join! 下で取り直す
  価値がある。
