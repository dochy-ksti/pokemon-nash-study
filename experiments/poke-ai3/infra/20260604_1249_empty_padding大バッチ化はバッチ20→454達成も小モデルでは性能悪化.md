# empty padding 大バッチ化: バッチ 20→454 達成も小モデルでは性能悪化

## 目的

推論バッチが平均 ~20 と極小だった問題を、empty padding + threshold ゲート (commit
15aa3ff) で解決した。実 GPU バッチが threshold 近くまで太ることは確認済み (別計測で
平均 217.7)。本実験は、この大バッチ化が実際の収束速度・スループットを改善するかを、
旧 early-flush 設計と同条件で比較する。

## 背景 (バッチが細かった原因と対策)

- 旧 root_task は「キューが一瞬空 → 閾値未満でも flush」していた。rollout が同時到達せず
  trickle するため早期 flush が多発し、実 GPU バッチ平均 ~20 (cap は num_games×W)。
- 対策: early-flush 削除 (threshold ゲートのみ) + 各 lookahead が常に W 本を計上するよう
  空きスロットを empty で埋める。empty は state=None で Python まで送り GPU から除外、
  request_id をエコーバックして ack。これでデッドロックも busy-spin もなく大バッチ化。

## コマンド

```bash
cd poke-ai3-python
uv run phase2-loop --num-games 32 --sim-concurrency 16 --sims 64 \
  --search-turn-min 4 --search-turn-max 8 --no-random --no-crit --stage 2a \
  --max-epochs 60 --checkpoint-path ../data/poke-ai3/phase2_lookahead_emptypad_ng32_w16.pt
```

- 環境: RTX 5090, CUDA。cap = num_games×W = 512。
- 旧設計の同条件は experiments/poke-ai3/20260603_2217 (num_games=32 W=16): 160s/60ep、収束 epoch 7。

## 結果

| | 旧 (early-flush) | 新 (empty-padding) |
|---|---|---|
| 実 GPU バッチ平均 | ~21 | 454 (max 512=cap) |
| 壁時計 (60ep) | 160s | 266s |
| 収束 epoch | ~7 | ~10 |
| 収束の正しさ | OK | OK (撃ち分け正常: vs_cloyster=1.0 / vs_goodra=0.0) |

- バッチは 20 倍以上に太った (21 → 454) が、壁時計は 1.66 倍 **遅く**なった。
- batch_stats: avg=454, max=512, ヒストグラムは 257-512 が 4481/4500。
- 収束の正しさは維持 (epoch 10 前後で correct_action_rate=1.0、撃ち分け正常)。
- exit 0、デッドロックなし。

## 考察 (なぜ遅くなったか)

- 総 real 推論数は両設計で同じ (同じ num_games/sims/epoch)。にもかかわらず遅い
  = この小モデル × RTX 5090 では推論は元々バッチサイズ律速ではない。
- 旧設計は小バッチを 32 ゲーム分パイプライン的に重ね、GPU を絶え間なく回して往復の
  固定オーバーヘッドを隠していた。
- 新設計は threshold ゲートで「512 件溜まるまで待つ → 一括 GPU → 応答待ち」という同期
  バリアを作り、パイプラインが浅くなった。加えて empty 往復のオーバーヘッド (ドレイン期に
  W-1 スロットが毎サイクル往復) が乗る。
- 小モデルは GPU 計算が一瞬なので、バッチを大きくしても計算は速くならず、バリア + empty
  コストだけが残った。

## 評価

ネガティブ (現状の小モデルにおいては)。「threshold まで GPU を駆動しない / 大バッチ化」
という目標自体は達成 (バッチ 454、デッドロックなし、収束維持) したが、性能はむしろ悪化。
この最適化が効くのは GPU 計算がボトルネックになる大きいモデルで、そこではバッチ化が計算を
償却して効くはず。小モデルでは旧 early-flush 方式 (高パイプライン・小バッチ) の方が速い。

## 次の選択肢

1. revert: 小モデルでは旧 early-flush が速いので戻す (empty-padding はモデル大型化時に再投入)。
2. 保持: 将来の大モデルを見据えて残す (小モデルでは遅いと割り切る)。
3. 中間: threshold を下げて (例 num_games×W の 1/4) バリアを浅くし、パイプラインと大バッチの
   折衷点を探る。empty 機構は残しつつゲートを緩める。

実装は commit 15aa3ff。
