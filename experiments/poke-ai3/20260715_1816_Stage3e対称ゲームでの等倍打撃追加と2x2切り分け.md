# Stage 3e 対称ゲームでの等倍打撃追加と 2×2 切り分け

## 目的

Stage 3c の各個体に `Crunch`（かみくだく、物理80）と `Dark Pulse`（あくのはどう、特殊80）を
追加した `Stage 3e` を作り、厳密 Nash がどんな形に落ち着くかを調べる。

Stage 3d は同じ操作を Stage 3b に施したものだった。3e を加えると 2×2 の格子が完成する。

| | 等倍打撃なし | 等倍打撃あり |
|---|---|---|
| 非対称 (Goodra-Hisui) | 3b | 3d |
| 対称 (Goodra) | 3c | **3e** |

これにより 3d 単独では答えられなかった問いが切り分けられる。3d で交代率が 55.1%→23.6% と
激減したのは、等倍打撃そのものの効果なのか、Stage 3b に元々あった非対称性の効果なのか。

## シナリオ

3e の各個体は次の3技を持つ。種族・実数値・チーム間の弱点技割当は 3c と同一である。

- Team1 Cloyster: Crunch / Dark Pulse / FightSpe60
- Team1 Goodra: Crunch / Dark Pulse / FairyPhy60
- Team2 Cloyster: Crunch / Dark Pulse / FairyPhy60
- Team2 Goodra: Crunch / Dark Pulse / FightSpe60

チームテキストは技以外 3c と byte 一致であることを確認した。技の宣言順は 3d と同じく
Crunch / Dark Pulse / coverage で固定する。`poke-env-rust` の行動 index は習得技スロット
相対なので、この順序が方策テーブルのスロット割当を決める（`nash_vi_cache.rs` の
`action_code` は MoveId から 0/1/2 を決めるため、両者が一致する必要がある）。
`cargo test` で `stage3e_stats_match_stage3c_exactly` を追加して実数値の同一性を保証した。

### ダメージ構造

両個体の Atk/SpA が同一（117/117）なので、ダメージは受け側の種族だけの関数になる。

```
Cloyster: maxHP=157 Atk=117 Def=222 SpA=117 SpD=94  Spe=90
Goodra  : maxHP=157 Atk=117 Def=94  SpA=117 SpD=222 Spe=90
```

| 受け | Crunch(物80) | DarkPulse(特80) | FightSpe60(特60) | FairyPhy60(物60) |
|---|---:|---:|---:|---:|
| Cloyster | 11-13% | 24-29% | 36-43% (SE) | 8-10% |
| Goodra | 24-29% | 11-13% | 8-10% | 36-43% (SE) |

3c では各個体が1技なので、不利対面では 8-10% しか通らず交代一択だった。等倍打撃を足すと
同じ個体が 24-29% を叩ける。さらに、居座られるなら coverage が最善だが交代されると 10% に
落ちるのに対し、反対種族用の等倍技は交代先に 29% 通るため、技選択が相手の交代を読む
マッチングペニー的な構造になる。3c にはこの構造が一切ない。

## 均衡計算

3c/3d と同じゲーム条件を使用した。

- HP buckets: 26
- 急所あり、16段階ダメージ乱数あり
- 毎ターン1%の幾何打ち切り (`discount=0.99`)
- 打ち切り時は生存数、次に残HP割合でタイブレーク
- cross-team only、全3,380,000有効状態

3e は最大4行動なので `solve_nash_cached_full` で `[Crunch, Dark Pulse, coverage, Switch]` の
完全方策を解く。3d 用に一般化済みのソルバが**無改修でそのまま使えた**（`action_code` が
`Crunch=0 / DarkPulse=1 / その他の技=2 / Switch=3` とステージ非依存なため）。

```bash
setsid nohup bash poke-ai3-python/scripts/run_3e.sh > /tmp/psro/run_3e.log 2>&1 &
```

ドライバが逐次実行する中身:

```bash
cd poke-ai3-python
uv run python scripts/run_nash_geo_h26.py --stage 3e --hp-buckets 26
uv run python scripts/run_nash_trajectory_compare.py \
  --stages 3c 3e --games-per-config 2000 --seed 20260715 \
  --output ../data/poke-ai3/nash_geo/trajectory_3c_vs_3e.json
```

本計算は393秒。遷移キャッシュは381,954,568 entries (~4.6GB)、VIは21層で収束した。

結果:

- exploitability mean: `0.00000184`
- exploitability max: `0.00009811`
- 全有効状態の平均方策:
  - Crunch 25.56%
  - Dark Pulse 24.81%
  - coverage 26.93%
  - Switch 22.71%
- 2行動以上を2%超で混合する状態: 2,293,321 / 3,380,000

### 整合検証

汎用の検証スクリプト `poke-ai3-python/scripts/verify_nash_table.py` を新規作成した。
方策の量子化和、控え瀕死時の交代確率、ゼロ和ミラー恒等式 `V(s)+V(swap(s))=1` を検査する。
このスクリプトが 3d の既存記録（mirror max=1 / mean=0.00794、exploit mean=0.00000195 /
max=0.00001514）を完全に再現することを先に確認し、スクリプト自体の正しさを担保した。

3e の結果（すべて ok）:

- 方策の量子化和: 全状態で 999〜1001
- 控え瀕死時に交代確率が非0の状態: 0件
- ミラー恒等式: 3,380,000 ペアで最大誤差1（実数0.001 = u16の1刻み）、平均0.00795
  （実数0.00000795）

3e は完全対称ゲームなので、このミラー検査は解の正しさを強く保証する。非対称な 3d では
同じ強度の検査ができなかった。

### 2×2 の切り分け

3b→3d と 3c→3e の効果はほぼ同一だった。

| 指標 | 3b→3d | 3c→3e |
|---|---|---|
| 交代率 | 55.1% → 23.6% | 54.1% → 24.5% |
| 平均ターン数 | 30.0 → 11.0 | 31.3 → 11.1 |
| 自然終局率 | 71.3% → 89.5% | 69.6% → 89.9% |
| 混合状態の倍率 | ×2.12 | ×2.06 |

したがって 3d で観測された交代の激減・対戦の短縮・混合の増加は、**Stage 3b の非対称性では
なく等倍打撃そのものの効果**である。非対称性を完全に取り除いても同じ数字が出た。

## 満タン開始局面

方策順は `[Crunch, Dark Pulse, coverage, Switch]`。

- Team1 Cloyster vs Cloyster（coverage が有効）: `V=0.584`, `[0.514, 0, 0.486, 0]`
- Team1 Cloyster vs Goodra（coverage が非有効）: `V=0.416`, `[0.388, 0, 0, 0.612]`
- Team1 Goodra vs Cloyster（coverage が非有効）: `V=0.416`, `[0, 0.388, 0, 0.612]`
- Team1 Goodra vs Goodra（coverage が有効）: `V=0.584`, `[0, 0.514, 0.486, 0]`

Team2 側は鏡像で、全4先発を等確率にした各チームの平均値はともに0.5になる。3c の対応する
局面は `V=0.587` で純粋戦略 `[1, 0]`（有利なら攻撃）/ `[0, 1]`（不利なら交代）だった。

満タン局面ではどの個体も等倍技を1本しか使わない。有利対面では coverage と「反対種族用の
等倍技」を混ぜ（相手の交代を読む）、不利対面では「反対種族用の等倍技」と交代を混ぜる。

## 支配される技についての当初の予測と、その反証

事前にダメージ表から次のように予測していた。FightSpe60 持ちの個体にとって coverage は
`(対Cloyster 43, 対Goodra 10)`、Dark Pulse は `(29, 13)`、Crunch は `(13, 29)` なので、
`0.55·coverage + 0.45·Crunch = (29.5, 18.5)` が Dark Pulse `(29, 13)` を両成分で上回る。
よって Dark Pulse は強支配され、均衡では使われないはずである（Crunch 持ちの側は対称に Crunch
が支配される）。

テーブル上で検証した結果、**この予測は 85.7% の有効状態でしか成立しなかった**。

```
予測「支配される等倍技」の確率が厳密に0: 85.747%
   平均確率 = 0.09736  最大 = 1.000
```

確率1で使われる状態すら存在する。使われる条件を調べると、相手の場の個体が低HPのときに集中
していた。

| 相手の場の個体のHP | 予測支配技が使われる状態の割合 | 平均確率 |
|---|---:|---:|
| ~10% | 32.6% | 0.2218 |
| 16-28% | 35.9% | 0.2732 |
| 32-52% | 11.4% | 0.0703 |
| 56-80% | 4.6% | 0.0265 |
| 84-100% | 2.9% | 0.0137 |

相手の控えが生存している状態では平均確率0.0992（使用状態14.6%）、控えが瀕死なら0.0520
（6.6%）だった。

原因は過剰ダメージ（overkill）である。相手が瀕死寸前なら coverage（43%）でも Dark Pulse
（29%）でもどちらもKOでき、超過分は捨てられる。すると差がつくのは交代先へ持ち越すダメージ
だけで、そこでは Dark Pulse（13%）が coverage（10%）を上回る。つまり Dark Pulse は
「KOしつつ交代先も削る」折衷として厳密に優位になりうる。

期待ダメージの線形比較による支配論法は、KO閾値という非線形性を無視しているため、満タン付近
でしか正しくない。満タン開始局面で Dark Pulse が 0 だったことがこの誤りを補強していた。

## 結論

対称ゲームでも等倍打撃の効果は非対称ゲームとほぼ同一で、交代率は半分以下、対戦時間は約65%
短縮、混合状態は約2.06倍になった。3d で見えた現象は Stage 3b の非対称性の副産物ではない。

また、完全対称ゲームであることを利用したミラー恒等式検査（最大誤差 u16 1刻み）により、
4行動 Nash ソルバの正しさを 3d では不可能だった強度で確認できた。

副産物として、「どのダメージ乱数でも劣って見える技が均衡で確率1で使われる状態がある」という
反直観的な現象を特定した。過剰ダメージにより、KO可能な局面では技の価値が「相手の場の個体への
ダメージ」ではなく「交代先へのダメージ」で決まるためである。

H=26テーブルと軌跡JSONは `data/poke-ai3/nash_geo/` に保存した。

### 追記2: 方策テーブルの u8 化（デプロイ失敗の修正）

配信サイトを確認したところ、**Stage 3d が公開されていなかった**。調査の結果、サイトは
3c時代のコミット `249b4eb` のビルドで凍結しており、3dを追加した `fdc7059` 以降デプロイが
一度も成功していなかった（配信中の `app.js` が `249b4eb` と完全一致することで確定）。
`/battle-3d.html` が 200 を返すのは Pages のフォールバックで、`/policy_3d.meta.json` は
JSONではなくHTMLを返していた。存在しないパスも 200 を返す。

原因は **Cloudflare Pages の 1ファイル 25MiB 上限**である。3d/3e の完全方策は u16×4行動で
27.89MiB になり上限を超える。上限超過のアセットが1つでもあるとデプロイ全体が失敗するため、
3dを入れた瞬間からサイトが固まっていた。3b/3c は P(交代) の u16×1行動で 6.97MiB なので
問題が無かった。

対策として完全方策を u8（scale 254、番兵 255）へ量子化し、27.89MiB → **13.95MiB** に
落とした。有効行の値は 0..254 に収まるので番兵 255 と衝突しない。行の和が厳密に 254 に
なるよう最大剰余法で丸め、確率が厳密に 0 の行動には増分を配らないので「控えが瀕死なら
交代確率 0」といった構造的な 0 は保存される。

量子化誤差は 3d/3e とも最大 0.0026、平均 0.00035 で、1刻み（1/254 = 0.0039）未満に収まる。
満タン開始局面の表示は 3e team0 が `51.4% → 51.6%` のように 0.2 ポイント程度動く。

- `web/app.js` は meta の `policy_dtype` を見て `Uint8Array` / `Uint16Array` を切り替える。
  3b/3c は u16 のまま据え置き（6.97MiB で上限に余裕があり、無損失を維持できるため）。
- exporter に 25MiB 超を弾く assert を追加した。
- `publish-web` スキルの整合検証手順を u8 対応に更新し、25MiB 検査を追加した。

ヘッドレスChromiumで 3d / 3e とも u8 テーブルで正常に描画・1ターン進行することを確認した。

ヘッドレスChromiumで日本語ページを検証した。表示された勝率・方策はNPZと完全一致した
（Team2 ヌメルゴン: 58.4% / `[51.4, 0.0, 48.6, 0.0]`、Team1 パルシェン: 41.6% /
`[38.8, 0.0, 0.0, 61.2]`、和は100.0%でミラー恒等式が画面上でも成立）。かみくだくを選んで
1ターン進め、ダメージ（相手ヌメルゴンに24%、自パルシェンに12%）が実測表と一致し、次局面の
方策がテーブルから再取得されることを確認した。3b/3c/3d の bin は今回の export で不変。
