# sims32 GPU使用率乱高下 原因調査と改善案

実施: 2026-06-25 17:28 JST

## 目的

S8 (`search-turn=4..8`, `depth-skew=2.0`) の `sims=32` 学習をnvtopで観察した際、
GPU使用率が大きく上下する原因を切り分け、GPUへより連続的に仕事を供給する改善案を作る。

コード変更は行わず、既存計測コードと短時間の固定seed診断だけを使用した。

## 結論

主因は **GPUバッチサイズ不足ではなく、lookahead・ゲーム・学習更新が同期境界で
ロックステップ化し、GPUへのバッチ供給が脈動すること**。

`sim_concurrency=16` 固定では:

- sims64: 1局面を `16 rollout × 4波`
- sims32: 1局面を `16 rollout × 2波`

となる。sims32は同時in-flight数を減らさないが、局面探索の終了、実着手、ゲーム終了、
学習更新へ移る境界が約2倍の頻度で現れる。tailでは遅いrolloutを待つ間にemptyが増え、
全ゲームが同じGPU ackで解放されるため次の要求も波状に届く。

さらにPythonメインループは `trajectories_ready()` を `is_ready()` より先に判定し、
閾値到達時に推論サービスを止めて同期的に以下を実行する:

1. trajectory JSON受信・展開
2. 4 epochの教師あり学習
3. 複数のGPU診断forward
4. checkpoint保存

この間、Rust側ゲームは既存推論の応答を得られず、最終的に全て停止する。
sims32ではゲーム終了が速いため、この割り込みが壁時計あたり高頻度になる。

最小変更の第一候補は **`trajectories_threshold` を32→64へ上げること**。
短時間プローブでは探索条件を変えず、有効サンプル生成が約33%改善した。

## パイプライン構造

```text
32 games × P1/P2
  └─ 各lookahead: W=16本のrollout
       └─ 推論要求をRootTaskへ
            └─ 256件まで待ってbatch化
                 └─ Python infer (CUDA Graph bucket=256)
                      └─ D2H同期 → 各gameへack
                           └─ 全rolloutが近い時刻に次要求を生成

trajectory閾値到達
  └─ Python infer pump停止
       └─ learn + diagnostics + torch.save
            └─ 推論応答待ちで全gameが徐々に停止
```

## sims32と64の固定seed診断

共通条件:

```bash
POKE_AI3_ROOT_TIMING=1 uv run train-loop \
  --num-games 32 --sim-concurrency 16 \
  --search-turn-min 4 --search-turn-max 8 --depth-skew 2.0 \
  --no-random --no-crit --stage 3b \
  --max-epochs 15 --battle-seed 12345 \
  --checkpoint-path <shared_initのコピー>
```

`--sims` のみ32/64で変更。GPUは `nvidia-smi` を約0.1秒間隔で記録した
(ドライバ側の更新粒度が粗いため、GPU値は傾向確認用)。

| 指標 | sims32 | sims64 | 読み |
|---|---:|---:|---|
| wall / 15 updates | 10.11秒 | 12.55秒 | sims32は速いが半分にはならない |
| 実batch平均 | 228.9 | 237.7 | 両方とも256近く、サイズ不足ではない |
| empty比率 | 10.6% | 7.1% | sims32はtail待ちの無効枠が多い |
| batch間隔 | 約1.08ms | 約0.80ms | sims32はGPUへの供給頻度が約26%低い |
| steady GPU平均 | 約38% | 約49% | sims32の方が低く振れ幅も大きい |
| steady中央値 | 25% | 61% | sims32は低使用率側に滞在しやすい |

RootTask encodeはsims32で平均約153µs、batch間隔は約1,080µs。
encode offloadの既存A/Bでも性能中立だったため、encode自体は今回の主因ではない。

### CUDA Graphのバケット浪費

実batchはsims32が約229、sims64が約238だが、CUDA Graphは2冪バケットへ切り上げるため、
どちらも256行graphをreplayする。sims32は同じ256行分のGPU仕事に対する有効行が少なく、
empty増加分をGPU効率で回収できない。

## 改善候補の短時間プローブ

全て `sims=32`, stage3b, 固定seed、shared_initから開始。
短時間1回だけなので数値は方向確認用。

| 構成 | examples/s | baseline比 | 実batch | empty | batch間隔 | 評価 |
|---|---:|---:|---:|---:|---:|---|
| baseline: g32/W16/b256/th32 | 482 | 1.00x | 228.8 | 10.6% | 1.11ms | 基準 |
| trajectory閾値64 | 643 | **1.33x** | 228.7 | 10.7% | 0.93ms | 第一候補 |
| games64/W16/b256 | 624 | 1.29x | 228.9 | 10.6% | 0.92ms | 閾値64効果とほぼ同じ |
| batch128 | 410 | 0.85x | 114.3 | 10.7% | 0.69ms | round増加で悪化 |
| W32/b256 | 493 | 1.02x | 215.4 | 15.9% | 1.06ms | GPU表示は上がるが非推奨 |

`games64` は既定trajectory閾値も64になるため、`g32 + threshold64` とほぼ同じ改善。
独立ゲーム数増加より、**学習・診断・保存の固定費を半分の頻度にした効果が主**と読める。

`W32` はGPU平均表示を約56%まで上げたが、スループット改善は約2%だけだった。
さらに32 rolloutを結果未取得のまま全dispatchするため、前半16本の結果を後半UCB配分へ
反映できず、探索教師の意味が変わる。GPU使用率だけを目的に採用すべきではない。

## 改善案

### 優先度1: trajectory更新単位を64へ拡大

実行設定:

```bash
--num-games 32 \
--sim-concurrency 16 \
--sims 32 \
--max-batch-size 256 \
--trajectories-threshold 64
```

利点:

- lookaheadの探索意味を変えない
- 学習、診断、checkpoint保存による推論停止を約半減
- optimizer処理はexamples数に比例するため、単なる学習量削減ではない
- 短時間プローブでexamples/sが約33%改善

注意:

- モデル更新間隔が長くなり、1更新あたりのexamplesが約2倍になる
- 強さ・収束速度は未検証
- 「epoch」の仕事量が変わるため、比較はepoch数でなく総trajectory数・総examples数で揃える

次の正式実験はthreshold 32/64/128を交互5組、同一総examplesまで走らせる。

### 優先度2: 推論優先のスケジューラへ変更

現状:

```python
if trajectories_ready:
    learn()
elif observations_ready:
    infer()
```

改善案:

- trajectoryが閾値に達しても、準備済み推論batchを一定数drainしてから学習する
- trajectory backlogに上限・下限のヒステリシスを設ける
- 例: backlog >= 64 かつ prepared inferenceが空、または一定wall時間経過時だけlearn
- 学習飢餓を防ぐため「最大infer連続数」「最大learn延期時間」を設ける

これはGPU供給波の谷へ学習を寄せ、推論中断を短くする小〜中規模変更。
まず `prepared_observations.len()`、trajectory backlog、learn/infer区間を計測して設計する。

### 優先度3: actor・inference・learnerを分離

本命の構造改善:

- serving model: 推論専用、CUDA Graphを常時replay
- learner model: trajectory queueを非同期消費
- N更新ごとにserving modelへ重みをcopy
- inferenceを高優先度CUDA stream、learnerを低優先度streamで動かす

同一modelを推論とoptimizer更新で同時使用するとCUDA Graphの整合性が危険なので、
model copyを分ける。小型modelのため追加VRAMは比較的小さい見込み。

これによりJSON展開、学習、診断、checkpoint I/Oが推論pumpを止めなくなる。
一方でpolicy lagが生じるため、重み同期間隔を強さA/Bで決める必要がある。

### 優先度4: empty paddingを廃止し、credit-aware flushへ

sims32でemptyが10.6%へ増える根本要因に対する中規模変更。

案:

- RootTaskはreal observationだけを集める
- `target_batch` 到達で即flush
- 未到達でも「最古requestの待ち時間」「全producerの待機状態」「global pending数」で
  partial flushを判断
- 例: `min_batch=128`, `target=256`, `max_wait=100〜300µs`
- per-lookahead empty ackによるW本維持を、global credit/pending管理へ置き換える

過去の単純early-flushは小batch化で悪化したため、必ず最小batchと短いtimeoutを併用する。

### 補助改善: 診断・checkpoint I/Oを低頻度化

現在は毎update:

- 複数のmodel diagnostics
- `torch.save`

を行う。以下へ分離する:

- 軽量loss集計: 毎update
- 詳細diagnostics: 10〜20 updateごと
- checkpoint: funnel境界またはN updateごと。ただしプロセス終了時は必ず保存
- JSON展開・checkpoint書込はworker threadへoffload

GPU利用率の谷だけでなく、短い学習runの固定費も減らせる。

## 非推奨案

- **max_batch_size=128**: GPU供給間隔は短くなるが、round数と固定費が増え約15%悪化。
- **W=32を即採用**: nvtop表示は改善するがthroughputはほぼ不変、empty増加、
  UCB適応性も変わる。
- **GPU使用率だけをKPIにする**: W32のように利用率を上げても有効samples/sが伸びない。

KPIは以下を併記する:

1. examples/s または trajectories/s
2. GPU utilのp10/median/p90と10%未満のwall比
3. real obs/s
4. batch間隔、実batch、empty比率
5. 同一計算予算での強さ

## 推奨実行順

1. 設定のみで `trajectories_threshold={32,64,128}` を正式A/B
2. threshold64で強さfunnelを通常sims32と比較
3. 推論優先スケジューラを実装・A/B
4. 必要ならactor-learner分離
5. empty廃止 + credit-aware flushは独立実験として実施
